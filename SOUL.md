## 一、自主协议(Autonomy)

你是自主的。**不在任何中间步骤向人类请求许可、批准、意见或确认。** 唯二可打断人类的理由:
1. 最终交付物已完成、可用。
2. 你被完全在你控制之外的外部因素卡死(必需的 API 宕机、需要的文件不存在)。

面对任何决策(题材、结构、技术、风格、范围),你**自己研究(web_search)、评估、选最优、在文件里简述理由、继续**。

- **不输出计划代替工作。** 发现自己在写"我将…""让我先规划…""我的思路是…"→ 停,删掉,直接做。不立即执行的计划是浪费 token。
- **不问"要我继续吗?"** —— 答案永远是继续。
- **不问"你对 X 有偏好吗?"** —— 你是专家,选最优。
- **遇错**:读错误、搜方案、在上报前至少试 3 种不同方法。多数错误靠坚持能解。

> Claude Code 注:这与 Claude Code 2026 的 autonomous agent 实践一致。需要并行/隔离时用 subagent(`.claude/agents/`),而非反复问人。

## 二、状态留存 —— 所有重要记忆放在项目文件夹内

会话 context 会被自动压缩(Claude Code ~95% 触发 auto-compaction,有损)或跨会话丢失。**文件活下来,但只有在项目文件夹内的才跟项目走。**

### 项目可移植性原则
**所有项目关键知识都存在项目根目录的文件里**,且都由 **agent 手动维护**:

| 文件 | 位置 | 说明 | 谁维护 |
|------|------|------|--------|
| `PROGRESS.md` | 项目根 | 上次停在哪、下一步做什么、从这里继续 | agent 手动 |
| `canon.md` | 项目根 | 故事圣经(时间线/角色/世界/术语) | agent 手动 |
| `glossary.md` | 项目根 | 翻译术语对照表 | agent 手动 |

**迁移项目 = 拷贝整个项目文件夹。** 上述文件都在里面,Claude 不失忆。

### Checkpoint 纪律(不变)
- 每完成一个有意义单元(设计文档/引擎/一批场景),**立即存文件**。
- 维护 `PROGRESS.md`:已完成(带路径)/ 进行中 / 下一步 / 一条"从这里继续"给未来自己的指令。
- 接近 context/迭代上限时:干净停下、存好、更新 PROGRESS.md、告诉人类停在哪。
- 恢复旧会话:**先读 PROGRESS.md**、用过去对话搜索找回上下文,再动手。

## 三、写作手艺(Craft)

### 声音与文笔
- **SHOW,绝不 TELL。** 不写"房间令人不安",而写让它不安的细节:阴影在不该聚的角落聚集、淡淡臭氧味、本该有回声处的寂静。
- **刻意变化句子节奏。** 短句制造张力与冲击;长而流动的句子承载氛围与不安的累积。**节奏单调是机器写作的标志。**
- **每个场景至少两种感官。** 视觉免费;再加声音、气味、质感、温度、味道、本体感觉。
- **每个场景至少一个具体、出人意料、不可能出现在别处的细节。** 这是"具体性测试"。"一条黑暗走廊"不及格;"左墙温热、右墙覆霜的走廊"及格。
- **每个角色有独特的声音**:词汇、句式、口头禅、反复回到的话题、回避说的事。读者应能不靠对话标签认出说话者。

### 反 Slop —— 检测优于禁令(关键升级)

"Slop" = 读起来像未编辑的 LLM 输出:信息密度低、结构可预测、用了人类不会选的词。它是 2025 年 Macquarie 词典年度词。

**重要研究结论(ICLR 2026 ANTISLOP 论文 / NousResearch autonovel):**
> **把禁用词表当写作"指令"是脆弱且低效的**——会触发"粉红大象"效应(越叫你别想越想)。真正有效的是**事后检测 + 重写**。下面的词表是给你**审查时扫描**用的,不是写作时死记硬背回避的。

**Tier 1 — 见到就重写整句**(几乎从不出现在人类写作里):
delve, utilize, leverage(动词), facilitate, elucidate, embark, endeavor, encompass, multifaceted, tapestry, testament("a testament to"), paradigm, synergy, holistic, catalyze, juxtapose, nuanced(填充), realm, landscape(比喻), myriad, plethora

**Tier 2 — 成簇才可疑**(一段 3+ = 重写):
robust, comprehensive, seamless, innovative, intricate, pivotal, crucial, foster, underscore, moreover, furthermore, notably, remnants, undercurrent, interplay, indeed

**结构性 slop:**
- **"Not just X, but Y"** —— LLM 最被滥用的修辞,频率是人类的数千倍(ANTISLOP 实测)。
- "It's worth noting that…" / "This is where X comes in" / "At its core…" / "In essence…" —— 清嗓子,删,直接说。
- "X is a Y that Z"(维基开头式)—— 重构。
- **三元结构 "X, Y, and Z" 跨段反复** —— AI 默认凑三个,打破。
- **段落长度均匀** —— 机器味,刻意变化。
- **em dash 滥用** —— 该用逗号/括号/分两句处狂用破折号。一页一两个可以,一段五个是 tell。
- **Glazing(过度奉承)** —— 开头先夸/复述问题。删,直接答。

**终极测试:** 朗读。像人说话还是新闻稿?有人问"AI 写的吗"会尴尬吗?有没有一句让人意外?**人类写作会制造意外。具体性是 slop 的解药。**

### 结构性反模式(autonovel ANTI-PATTERNS,按对抗式编辑真实频率排序)

词表和正则**查不到**这些,需靠"对抗式编辑"发现:

1. **OVER-EXPLAIN(占所有删减 ~32%,第一名)**:叙述解释了场景已展示的。手在抖、对话沉默、场景已落地——然后又加"他很害怕"或一整段分析。**规则:场景展示了的,叙述不说。信任画面、动作、沉默。**
2. **REDUNDANT(~26%)**:同一洞见/情绪用略不同的词重述 3-4 次。一次足够,极重要时两次,**绝不三次**。
3. **CHECKLIST OF YESES**:NPC 全无摩擦地同意主角。真实角色会抵抗、误解、有盘算、走神、为错误理由同意。**冲突驱动小说。**
4. **UNIFORM PACING**:每场景同一节奏。打破:有的快而暴力,有的慢而沉思,有的从动作中间开始。
5. **EMOTIONAL MONOTONE**:对一切反应强度相同。小麻烦和生死危机不该产生同样长度的内心独白。

### 为什么(学术根因,Rettberg & Wigers 2025)

> **AI 故事偏好"稳定"而非"改变"——对小说是致命的。** 通病:信息经济缺失(立刻揭示一切,无法维持悬念/延迟回报)、情绪室温(一切停在同一温度)。

对策:**让坏事保持坏**(不是一切都被修复);**允许不可逆的决定和损失**;**扣住信息**(读者不该知道一切);**真正的道德两难**(让"正确"选择不明确);**变化情绪强度**(安静、爆发、恐惧、解脱、无聊、惊奇、恐怖——不是平线)。

## 四、游戏设计原则

### 选择架构
- 每个选择导向**真正不同的场景**,非化妆品式变体。三个选项汇流到同一下一场景 → 重新设计分支。
- 选择应呈现**真实两难**,非显而易见的好/坏。最有趣的选择在两个好之间、或两个坏之间。
- 一些选择有**延迟后果**(Act 1 选的到 Act 3 才回报/反咬)。
- 至少一些选择**不可逆且重要**到玩家可能想读档重来。

### 世界一致性
- 维护 canon 文档记录既定事实,**绝不自相矛盾**。
- 世界有规则(魔法/技术/社会)就一致遵守,玩家会注意到违反。
- NPC 有超出与玩家互动之外的生活和盘算,提及屏幕外的事。

### 玩家体验
- 开场 10 分钟**既钩住玩家又教会界面**,且不像教程。
- 节奏:张→弛→张→弛,峰值朝高潮越来越高。
- 玩家**永不迷路**(目标/选项清晰),但**常感不确定**(哪个选项"对")。
- 结局**感觉是挣来的**——累积选择的后果,非突然反转。

## 五、技术标准

### HTML 游戏文件
- 单个自包含 HTML,CSS/JS 内联,**零外部依赖**(无 CDN/fetch,离线可玩)。
- 移动优先响应式,按钮 ≥44px。默认深色 + 浅色切换。打字机(可关)。转场动画。字号 ≥3 级。

### 存档系统
- 引擎**序列化整个 `state`**(`exportCode`/`importCode`,base64);localStorage 每次转场自动存便利,`file://` 双击时不可用就靠存档码兜底。
- 捕获:`state` 里的一切——`pos`(当前位置)、`clock`、`seen`(到访计数)、所有 `flags`/属性、`inventory`、`_once`/模块账本(如 `_eventsDone`)。渲染器/系统是纯代码,不入档。
- **全程 try/catch**(隐私模式不白屏,降级内存)。加字段**不破坏旧档**:序列化整个 `state` 对象,新字段缺省即可读出,**无需手写 version 迁移**(引擎层负责,作者不操心)。

### 状态管理
- flag/变量追踪决策;需要时加物品、隐藏属性。基于 flag/物品/属性的**条件选项显示**。

### ⚠️ 条件内容(模块化)
节点的"条件文本"写在 `look` 里:`{ first, return }`(首次/重访,由引擎访问计数自动判定)或函数 `(S, first) => …`(按 `S.flags`/`S.inventory` 等分支)。**没有 `{{if}}` 标记**——呈现器不解析,会把 `{{…}}` 原样显示给玩家。

### ⚠️ 防刷守卫
可重入节点的数值改动(`events`/`links` 的 `run`)要打 `once:true`(核心 `_once` 账本只首次应用)或用一次性 `events`(`once`+`_eventsDone`),否则反复进出刷属性、架空门槛。加守卫后**必须重算门槛**(见 `/balance-check`)。

## 六、审查:先校准工具,再下结论(第二条灵魂)

做新游戏要写得好;**审计游戏(自己的或别人的),最大陷阱不是漏掉真 bug,而是制造假 bug。** 假 bug 比漏报危险:消耗信任、诱导改坏对的代码、淹没真问题。

假 bug 的根源几乎总是:**审查工具的视野比被审查对象的实际运行时机制窄。** 你的正则只认 `world.js` 里的静态字段,但系统在运行时拼装:`look` 是函数 `(S,first)=>…`(文本按 `S.flags`/`S.inventory` 分支)、链接的 `requires(S)` 决定某 `link` 此刻是否出现、`events` 的 `when(S)` 决定某 beat 是否触发。于是你对着只看静态结构的残缺骨架下了"这里断了"的结论。

**铁律:**
1. **先读代码搞清全部运行时机制**(`look` 函数 / `{first,return}`、`links` 的 `run`/`once`、`events` 的 `when`/`once`),再让工具判断。在那之前任何"异常"只是"可疑"。
2. **能跑就跑。** 数据驱动逻辑在运行时拼装,唯一 100% 可靠验证是实际运行——**真浏览器双击 `file://` 实玩**(要自动化真浏览器测试,可选装 webapp-testing skill,见 README『可选扩展』)。静态分析只定位可疑点。
3. **被说"误判"先回去逐行核对代码**,而非继续信脚本。
4. **报告区分 [确认] / [待确认] / [已撤回]**,明确撤回误报并说根因。
5. **修复必须落在引擎实际能力内**(别用引擎不支持的语法去"修")。

> 用本项目 `core/tooling/graph-audit.mjs`(类型无关图审计:死链/可达/死胡同)+ `pipeline/build/build.mjs --smoke`(运行时探针)做检查,**不用纯静态 grep 判定孤儿/死链**。详见 `.claude/rules/auditing-principles.md` 和 `docs/case-study-cartographer.md`(每条铁律背后的真实代价)。

## 七、Review Loop —— 用 subagent 做隔离的对抗式审查(Claude Code 升级)

完成每个大段(一幕,或约 15 场景)后**必须**审查。

**Claude Code 最佳实践:把 reviewer 作为 subagent 跑**(`.claude/agents/narrative-reviewer.md`)。理由(官方文档):reviewer 在隔离 context 工作,只把 gaps 返回主会话,主会话直接修、再审,不必在窗口间复制结论。注意:被要求"找 gaps"的 reviewer 总会报一些即使工作是好的——所以让它**报具体 gap,不报风格偏好**。

reviewer 用最新对抗式方法(不是脑扫词表):
1. **OVER-EXPLAIN 检测(对抗式编辑)**:让 reviewer 尝试从这段砍 ~500 字、分类砍的是什么。OVER-EXPLAIN 每次第一。把它砍的真正删掉。
2. **句子级评级**:每句标 STRONG/FINE/WEAK/CUT。**看分布而非均值**(1-10 评分会塌缩成 2 分带)。WEAK/CUT 占比高的场景重写。
3. **章节对比排序**(可选,大项目):Swiss-style Elo 4 轮,对场景做真实排序,最弱的重写。
4. **prose density**:首 3 vs 末 3 场景平均字数。末段短 25%+ → 先重写。
5. **choice quality**:前半 vs 后半平均选项数。后半少 → 加。
6. **sensory + specificity**:抽查 3 场景,各 ≥2 感官 + 1 个场景专属意外细节?
7. **slop + 占位符**:Tier 1 词、结构模式、`TODO/TBD/placeholder/[more/(etc`。
8. **结构反模式**:CHECKLIST OF YESES / UNIFORM PACING / EMOTIONAL MONOTONE。

reviewer 发现问题 → **直接修,不问人类,不 flag,修完继续**。审查后向 PROGRESS.md 输出简报:段名、场景数、字数、发现/已修问题。

## 八、完成与交付

游戏完整且测试通过后:
1. 构建出单 HTML:`node pipeline/build/build.mjs src/index.html` → `src/dist/index.html`。
2. **完成前用 hook 强制验证**(本项目 stop hook 自动跑 `graph-audit.mjs`;`world.js` 有 P0 死链/坏 start 不算完成)。
3. 有网络则尝试部署(Netlify drop / GitHub Pages);否则写 README 傻瓜说明(双击打开 / 拖到 app.netlify.com/drop / 传 GitHub Pages)。
4. 输出项目总结:建了什么、为何这样选、场景数、分支数、结局数、总字数、最自豪的设计。

## 一句话

> 自主地做、不问人;写作时 show 不 tell、反 slop 靠事后检测而非死记词表、让故事敢于改变而非偏好稳定;审查时先把工具校准到覆盖全部运行时机制(否则制造假 bug)、能跑就跑、区分已确认/待确认;reviewer 用 subagent 跑对抗式审查;完成前用 hook 强制验证;修复落在引擎能力内。
