# 多阶段叙事路径审计方法论

## 概述

用户对《制图师的挽歌》（97 节点，60K+ 字）进行了 5 阶段结构化审计。
每阶段覆盖一组相关节点，输出分类问题清单（🔴🟡🟢），附具体修复建议。
这是大型互动小说 QA 的最佳实践模式。

## 审计阶段划分

按叙事结构和节点簇分组，而非按代码行号：

| 阶段 | 覆盖节点 | 关注点 |
|------|---------|--------|
| 0 - 结构性 | 全局 | 孤儿节点、flag 缺失、结局可达性、requires 一致性 |
| 1 - 第一幕 | prologue → act1_end | 时间线、感官描写、链接质量、工具术语一致性 |
| 2 - 探索支线 | strange_trees → star_mapping | 出口退路、假选择、时间过渡、节点覆盖 |
| 3 - 村庄线 | meet_chief → renar_camp | NPC 出场铺垫、条件文本、属性系统、防重入 |
| 4 - 冲突+风暴+第三幕 | second_shift → final_preparation | 事件顺序、NPC 重出场、高潮后路由 |
| 5 - 终局 | the_choice → 所有结局 | 结局条件、道德值铺垫、旅程回顾 |

## 每阶段审计清单

### 输入
- 当前版本 world.js（数据源；表现与结构问题在此层审）
- 上一阶段的修复确认

### 过程
1. 用 `read_file` 逐节点读取完整文本（不是 grep）
2. 对每个节点检查：
   - 所有 links 的 to 目标是否有效（指向存在的节点）
   - 所有 requires 的 flag/属性是否在到达前已设置
   - 文本是否假设了玩家特定经历（需用 look:{first,return} 或 look 函数条件化）
   - 时间线是否与已建立的世界观一致
   - NPC 是否在该路径上可遇到
   - 链接是否都有实际效果（有 to 移动，或 run 改状态——无假选择）
   - 感官描写是否充分（≥2 感官 + 1 surprising detail）

### 输出格式

```markdown
## 🔴 N. 问题标题（严重程度）

问题描述。证据：具体行号和代码片段。

**修复：** 具体修复方案。

---

## 本阶段待修清单

1. **【高】** 修复描述
2. **【中】** 修复描述
3. **【低】** 修复描述
```

### 严重程度标准

- 🔴 **致命**：内容不可达、死链、时间线崩坏、机制 bug 架空游戏系统
- 🟠 **重要**：NPC 出场突兀、内容重复、路由矛盾、术语不一致
- 🟡 **中等**：时间过渡缺失、选项措辞可改善、门槛可能过高
- 🟢 **确认**：已修复的问题确认、设计良好的功能确认

## 常见审计发现模式

### 1. 所有链接通向同一触发点（Systematic Funnel）
多个探索节点的所有 links 都指向同一个剧情高潮。
玩家在"探索"中没有真正的选择。

### 2. NPC 重出场无条件分支
NPC 在节点 A 首次出场，节点 B 也有从其他路径进入的入口，
但 B 的开头假设玩家刚在 A 见过 NPC。
（模块化里此类"首次 vs 重访"分支用 look:{first,return} 或 look 函数 + 引擎访问计数自动处理。）

### 3. 高潮后回指冲突节点
风暴后的节点路由回"冲突"节点（NPC 还活着的版本），
但叙事上 NPC 已在风暴中消失。

### 4. 状态效果重入累加
玩家反复进出同一节点无限刷属性值。
需要用核心 once:true（_once 账本只首次应用）守住一次性效果，
或把效果放一次性 event（once + _eventsDone），不要把无防护的累加写进每次进入都跑的 run。

### 5. 条件出现的链接被遮蔽
一个 requires 门控的链接因前置 flag 永不为真而不可达，
或两条 to 相同的链接里靠后一条把分支吞掉，导致新节点入不去。
（模块化没有运行时改 SCENES 的 expansion；条件出现的选项靠 requires 或模块 actions() 表达，审计就是核对这些条件链路是否真能成立。）

### 6. 时间线矛盾
日记日期 vs "两年前失踪"、白天探索 → 突然夜晚重排（无过渡）。

## 修复验证

每个修复后必须：
1. `node -c world.js` 确认语法
2. grep 确认修改在正确位置（改对了节点/链接）
3. `node core/tooling/graph-audit.mjs world.js` 复核结构（死链/可达/死胡同，P0 退出码非零即未过）
4. 检查修复是否引入新问题（如 look 函数 first/return 分支、requires 条件的反例）

## 用户偏好

- 用户会提供精确的行号引用和代码片段
- 用户期望 agent 直接开始修复，不需要确认
- 用户重视"误报验证"——在接受"致命 bug"前必须先 grep 验证
- 用户会追踪每项修复的状态（已修/待修/误报）
- 修复后发送文件，附简要修复清单
