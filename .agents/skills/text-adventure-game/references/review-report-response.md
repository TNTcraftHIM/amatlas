# 通关测评报告响应方法论

## 概述

当用户（或外部测试者）提供结构化的通关测评报告时，
按以下流程处理。报告通常包含：总体印象、按优先级排列的问题列表、
确认无问题的方面。

## 处理流程

### 1. 先验证，再修复

**不要盲目相信报告中的每个断言。** 报告可能是基于旧版本代码写的，
或者测试者的理解有误。

**必须验证的项目：**
- 读 world.js 确认报告中提到的问题是否仍然存在于当前数据
- 检查报告中说"已修复"的是否真的已修复
- 检查报告中说"未实现"的功能是否实际已有实现

**典型误报（false positive）：**
- 报告说"六代制图师顺序不一致"——实际所有场景都是一致的顺序
- 报告说"cave/village/heart 音频未实现"——实际所有 region 的 audio 意图全部已声明
- 报告说"requires 未使用"——实际链接已有递进 requires 设置

**验证方法：** 读 world.js / 用 graph-audit 逐项确认，不要凭记忆判断。

### 2. 优先级分类

报告通常自带优先级。按以下顺序处理：

| 优先级 | 类型 | 修复策略 |
|--------|------|----------|
| P0 | 引擎 bug（崩溃/死链） | 立即修 |
| P1 | 玩家直接察觉（假选择/误导选项） | 立即修 |
| P2 | 叙事逻辑（时间线/语气冲突） | 修 |
| P3 | 技术确认（audio/scene 意图/region） | 验证 + 修 |
| P4 | 文字打磨（"一种"减量/开头多样性） | 批量修 |
| P5 | 可选改进（墨水经济/悬念回收） | 记录，按需修 |

### 3. 假选择差异化

报告中常见的"假选择"问题：同一节点 2+ 个 link 指向同一 `to`，且 `run` 无区别（无 run、或 run 体等价）。

**修复策略（不要合并为单一 link——保留选择感）：**
- 给其中一个 link 的 `run` 加独特 flag：`run: (S) => { S.flags.recorded_shift = true; }`
- 让两个 link 的属性增减不同：一个 `run: (S) => { S.understanding = (S.understanding||0) + 2; }`，另一个 `run: (S) => { S.mapProgress = (S.mapProgress||0) + 5; S.ink -= 1; }`
- 在 link 的 `label` 文本中加入不同的情感色彩（"记录"vs"观察"vs"等待"）

**效果分类（都写进 `run` 函数体改 `S.属性`/`S.flags`）：**
- "记录/画图/用墨水" → `S.mapProgress += N` + `S.ink -= N` 消耗 + 可能的 `S.flags.X = true`
- "观察/思考/等待" → `S.understanding += N`
- "跟随NPC/合作" → `S.relationNative += N`
- "独立行动/大胆" → `S.moralControl += N` 或 `S.moralTruth += N`

### 4. 批量文字替换技巧

当报告指出高频用词问题（如"一种"出现 400+ 次）时：

**不要逐个手动 patch。** 用一段 Node 脚本（`fs` 读写 + 批量 `split/join` 替换）处理 **world.js 源数据**（不是构建产物 HTML——改源、再 `node pipeline/build/build.mjs` 重建）：

```js
const fs = require('fs');

const filepath = "/path/to/world.js";
const replacements = [
  ["一种你无法辨认的X", "辨认不出的X"],
  ["一种你无法命名的X", "无法命名的X"],
  ["一种你无法完全理解的X", "说不清的X"],
  ["带着一种X的气味", "带着X的气味"],
  // ... 更多模式
];

let text = fs.readFileSync(filepath, "utf8");
let count = 0;
for (const [oldStr, newStr] of replacements) {
  if (text.includes(oldStr)) {
    text = text.split(oldStr).join(newStr);  // 全部替换（非仅首个）
    count += 1;
  }  // 否则：已被替换或未找到
}
fs.writeFileSync(filepath, text);

console.log(`Applied ${count} replacements`);
```

**可安全替换的模式（约占 30%）：**
- `一种你无法X的Y` → `无法X的Y` 或更具体的描述
- `一种X的感觉` → `X的感觉`
- `一种X的气味` → `X的气味`
- `一种X的声音` → `X的声音`
- `一种从未有过的X` → `从未有过的X`
- `形成了一种X` → `形成了X`

**不应替换的模式（"一种"是语义必要的）：**
- `一种新的东西`（强调"新类型"）
- `一种流动的记录`（定义新概念）
- `创造一种新的记录方式`（强调独特性）
- `一种选择`（表示"a kind of choice"）

**目标：** 从 400+ 降到 300 以下（减 25%+）。
**验证：** `grep -o "一种" world.js | wc -l` 确认数量。

### 5. "你"开头场景比例检查

第二人称游戏里"你"开头的场景偏高时，节奏感会重复。

**检测：** 在 world.js 里数 `look` 开头是"你"的节点数 / 总场景数（如 `grep -c "look:.*'你" world.js`，函数式 look 需读首句判断）
**目标：** < 35%
**修复策略：** 把一部分改成以环境/声音/光线开头：
- `你醒来的时候，嘴里全是沙子` → `沙子的颗粒感在舌头上散开`（保留第二句再接"你"）
- `你走进了森林` → `紫色的树冠下，光线变成了深蓝色`

### 6. 结局条件文本基调审查

结局节点的 `look` 常按 flag 分支出不同文本段（模块化里：`look: (S) => S.flags.rechecked_case ? '…' : '…'`）。
对每个结局 `look` 里的每个条件分支段：
1. 读完条件分支段 → 记录情绪方向（自信/后悔/平静/恐惧）
2. 读完结局正文 → 记录整体基调（悲剧/胜利/中性）
3. 问：条件分支段的情绪方向是否与结局基调一致？

**不一致的修复策略：**
- 悲剧结局 + 正面条件分支段 → 改为反讽口吻
- 胜利结局 + 负面条件分支段 → 改为克服困难的描述
- 中性结局 + 极端条件分支段 → 改为沉思/理解的描述

**实际案例：** ending_control 节点 `look` 里 `S.flags.rechecked_case` 的分支段说
"你不会重蹈覆辙，你会做得更好"——但结局正文是"代价是岛屿的生命"。
修复为反讽："你读过了每一个字。然后你做了和他一样的选择——
只是你告诉自己，这次会不一样。"

### 7. 时间线矛盾检测

建立权威时间线后，搜索所有涉及时间的描述：

```bash
grep -n "很新\|最近\|刚.*过\|刚刚\|没有愈合" world.js
grep -n "1894\|1895\|1896\|1897\|两年前\|一年前" world.js
```

**修复原则：** 如果角色 X 年前就离开了，其留下的痕迹
必须描述为"已经风化/模糊/被苔藓覆盖"，不能是"很新/刚留下的"。

### 8. 资源经济重平衡

当报告指出"资源经济没有压力"（设定为稀缺但实际用不完）时：

**审计步骤：**
1. 用 regex 提取 world.js 里所有 `S.ink += N` / `S.ink -= N` 模式（在 link 的 `run` 与 event 的 `run` 里）
2. 分类为 gains（`+=`）和 losses（`-=`）
3. 计算：初始量 + 总获取 = 最大可用，总消耗 = 最大消耗
4. 计算典型路径（10-15 消耗 + 5-8 获取）净额

**平衡目标：**
- 典型路径净额 ≈ 0（刚好够用，玩家必须选择）
- 全做净额 < 0（不可能全做，强制取舍）
- 初始量给"前 3 个选择"预算，不给"全程"预算

**调整策略（按效果从大到小）：**
1. 初始量减少（如 10→6）——立即感受到压力
2. 最大单次获取减少（如墨泉 +5→+2）——补给不再"免费"
3. 准备/过渡场景获取减少（+3→+1）或删除（+1→0）
4. 关键记录消耗增加（-2→-3）——每次蘸墨都有代价
5. 结局消耗保持不变（-5 是有意义的终局承诺）

**关键注意事项：**
- 初始值在 **一处** 定义——模块/world 的 state 初始（如模块 `init` 返回的初始 state，或 world 的 state 种子），改这一处即可（模块化引擎序列化整个 state 存读档，**无手写的 restart/continue 默认值要同步**）
- 某些获取有"link `run`"和"event `run`"两个入口（如墨泉 +5 既是某 link 的 `run`，也是进入场景的 event `run`）→ 两处都要改
- `replace_all=true` 适合全局替换同一值，但要确认没有误伤（如 `S.ink += 3` 与某处 `S.ink -= 3` 不应被一并改动）

**验证：** 重新运行审计（按步骤 1 的 regex 重数），确认典型路径净额 ≈ 0。

### 9. 同一节点的"条件出现"链接重复陷阱

模块化里没有运行时改写场景的机制（不存在旧的 `SCENES["id"].text = …` / `.choices = […]` 覆盖）。
"条件才出现的选项"靠 link 的 `requires:(S)=>bool`（配 `showWhenLocked`/`lockHint`），或放进模块 `actions()`。
但同一节点里可能 **既写了无条件 link，又写了带 `requires` 的同 `to` link**，
等价于旧时代"原始定义被 expanded 覆盖"的隐患——两条都生效时门控形同虚设。

**审计陷阱：** 审计报告读到无 `requires` 的那条 link 报问题 → agent 给它加了 `requires` →
但同节点另一条指向同 `to` 的 link 仍无门控 → 玩家照样能无条件通过。

**实际案例：** act1_end 节点的某条 link 无 `requires`，另一条同 `to` 的 link 才有
`requires: (S) => S.flags.arrived_village`。报告说门控是问题 → agent 只修了一条 →
另一条仍无门控 → 门控仍然失效。

**验证方法：** 修复后必须读该节点 **所有指向同 `to` 的 link**，确认门控一致：
```bash
# 检查某门控字段在该节点所有同 to 链接中的状态
grep -n "arrived_village" world.js
grep -n "act1_end" world.js  # 找该节点及所有引用它的 to
```

**修复策略：**
- 删掉重复的无门控 link（保留带 `requires` 的那条），或
- 给所有指向同 `to` 的 link 加同样的 `requires`
- 用 `replace_all=true` 当多条需要同样修改时

### 10. 多轮迭代中的值同步问题

模块化里资源初始值理应只在 **一处**（state 初始）定义，迭代中要警惕意外的重复硬编码：
1. state 初始（如 `{ ink: 10, … }`，新游戏初始值）——唯一权威源
2. 某节点 `run` 里误把初始值当常量再写一遍
3. 文案里提到的数字（"你有 10 滴墨水"）与初始值脱节

**铁律：改初始值时，grep 所有出现位置，确认只有权威源该变、其余是否需同步。**
```bash
grep -n "ink: 10\|ink = 10\|10 滴墨水" world.js  # 找所有硬编码的旧值
```
`replace_all=true` 适合这种情况，但要确认没有误伤（如 `S.ink += 10` /
`S.ink -= 10` 这类 `run` 体里增减量中的 10 不应被一并改动）。

## 常见陷阱

1. **报告可能有误报** — 验证每个断言再修复
2. **批量替换可能破坏语法** — 每批替换后 `node -c world.js` 验证
3. **"一种"替换可能改变语义** — 保留语义必要的实例
4. **`look` 条件分支语气冲突不是引擎 bug** — 是叙事设计问题
5. **报告中的"可选改进"不必全部实施** — 按优先级和影响决定
6. **资源经济调整时值可能被重复硬编码** — grep 确认权威源外的实例同步修改
7. **同节点重复链接令门控失效** — 修复后读节点所有同 `to` 链接确认门控一致
8. **分阶段审计报告可能引用旧版本** — 每阶段开始前重新验证前几阶段的修复是否仍有效
