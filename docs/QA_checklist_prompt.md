# 分支叙事游戏 · 通用审查要求（Prompt）

> 用途:审查 Amatlas 数据驱动的分支叙事游戏(文字冒险、视觉小说、互动小说等),尤其是 **LLM/agent 生成**的此类游戏。Amatlas 游戏 = `world.js`(数据:`maps→nodes`,每个节点有 `look` 正文 + `links` 选项)+ `game.js`(组装)+ `index.html`(模板),经 `node pipeline/build/build.mjs` 构建成单 HTML。每个选项(link)带跳转目标(`to`)/ 门控(`requires`)/ 状态转移(`run`),首次/重访由引擎访问计数自动判定。
>
> 这份清单是从一个真实项目(约100场景、6结局、10万字的中文文字冒险)的多轮审查中提炼的,**每一条都对应一个真实踩过的坑**。按"先搞对方法,再查机制,再查状态,再查叙事,最后查内容与可玩性"的顺序组织。形状以 `examples/text-adventure-demo/world.js` 与 `core/module-interface.md` 为准。

---

## 第 0 层:先校准你的审查工具(最重要,血泪教训)

**在报告任何"bug"之前,先确认你理解了引擎的全部运行时机制。** 静态扫描 / 正则解析极容易漏掉运行时构造,而**解析器的盲区会变成你口中的假 bug**。这个项目里,我曾因为解析器只认静态选项定义、不认运行时根据 state 决定的逻辑,连续多轮误报"孤儿场景""flag 未设置""结局锁死",最后逐行读代码才发现全是误判。Amatlas 的数据驱动模型把这类盲区大幅收窄(没有运行时改数据的机制),但**函数式 `look`/`requires`/`run`、模块 `actions()` 动态产出的选项、`events` beat** 仍是静态正则看不全的运行时逻辑——务必读 `world.js` + `game.js` + 模块 `runtime/` 一并理解。

动手前必须先在代码里确认以下机制是否存在、如何工作:
- [ ] 选项除了节点静态 `links: [...]`,模块 `actions(state,node)` 是否还**按 state 动态产出**额外选项(如条件出现的动作)?Amatlas 没有运行时改数据的机制,条件出现的选项要么放模块 `actions()`,要么用 link 的 `requires`。
- [ ] 进入节点的副作用:节点 `events:[{when, run}]` beat(进入即触发,常用来设 flag),以及函数式 `look:(S,first)=>…` 里读的状态——这类最容易被漏看。
- [ ] `look`/`requires`/`run` 是否为**函数形**(运行时根据 state 生成正文 / 决定门控 / 改状态)?结局分流常用 `look:(S)=>…` 按 flag 选分支。
- [ ] 首次/重访由引擎**访问计数**自动判定(`look:{first,return}` 或 `look:(S,first)=>first?…:…`;`firstTime`=`visits<=1`),不是手写 flag——确认作者没有自己重造一套。
- [ ] `game.js` 是默认 `Amatlas.boot(WORLD, manifest)` 形态,还是手写 `createEngine + engine.use(...)` escape hatch?列出 manifest / 插件 / presenter;有没有插件改变可达性或状态(一般不会,但要看一眼)。

**铁律:**
1. 把上述所有机制纳入你的解析器后再判断可达性 / flag / 死链。优先用引擎自带审计:`node core/tooling/graph-audit.mjs world.js`(死链 / 可达 / 死胡同;P0 退出码非零),它已把图结构机制纳入。
2. **能跑就跑。** 数据驱动游戏的逻辑在运行时拼装,唯一 100% 可靠的验证是**在浏览器/运行环境里实际玩**(`node pipeline/build/build.mjs src/index.html --smoke` 用 jsdom 探针加载一遍构建产物)。静态分析只能定位"可疑点",不能下"确凿 bug"的结论(函数式 `look`/`run`、视觉/音频意图尤其如此)。
3. 当开发者/对方说"你报的 bug 是误判"时,**优先回去逐行核对代码**,而不是继续相信自己的脚本。
4. 报告里**区分"已确认"和"待确认"**:能跑通验证的标"已确认",只靠静态推断的标"待确认/需运行验证"。

## 第 1 层:引擎与机制正确性

### 1.1 条件正文写在 `look` 里(模块化呈现器不解析任何 `{{}}` 标记)
- [ ] **首要确认**:Amatlas 的呈现器(`present-dom.js` 等)**不解析任何 `{{}}` 标记**——`look` 字符串里写 `{{if:...}}` 会被**原样显示**给玩家(可见崩坏)。条件正文应写在 `look` 的函数/对象形里,用真正的 JS 逻辑:
  - 首次/重访:`look:{first:'首次版', return:'重访版'}`(引擎按访问计数自动选)。
  - 按 flag/属性分支:`look:(S,first)=> S.flags.X ? '甲' : '乙'` 或 `look:(S,first)=> first ? '初到…' : (S.flags.X ? '…' : '…')`。
- [ ] 全文搜索 `world.js` / 内容文件里**残留的 `{{...}}` 标记**(从旧格式迁移时最易漏):`{{if}}`、`{{else}}`、`{{endif}}`、`{{elif}}`、`{{else if}}`、`{{if:inventory}}`、`{{if:ATTR>=N}}`。**任何 `{{}}` 出现在 `look`/正文字符串里 = 玩家会看到原始标记**。改成 `look` 函数里的对应 JS 判断(flag→`S.flags.X`;物品→`S.inventory.includes('X')`;属性→`(S.attr||0)>=N`)。
- [ ] 确认条件判断用对:布尔事实读 `S.flags.X`(真假);需要按"取值"分支时,JS 里 `S.flags.X === 'value'` 完全可写——但更清晰的惯用法仍是给每个取值单独设一个布尔 flag(便于 `requires`/审计对齐)。`look` 是普通 JS,**没有"引擎不支持的语法"这种崩坏**;真正的风险只剩"把条件误写成 `{{}}` 字符串"。

### 1.2 重访节点的属性累加要有守卫
- [ ] 检查"进入节点即跑"的 `events:[{run}]` beat:**纯累加数值属性的 beat 必须带 `once:true`**(引擎据 `_eventsDone` 账本保证只触发一次),或用 `when:(S)=>!S.flags.gotX` 自门控。无守卫且节点可重入(见1.3)= 玩家**反复进出刷属性**。Amatlas 的 `once:true`/`_once` 账本随档,是这道守卫的现成机制。
- [ ] 检查 beat 的 `run` 对数值属性是否**纯累加、无上限、无去重**。若是、且无 `once`/`when` 守卫、节点又可重入,则可被刷爆。
- [ ] 区分**进入级 beat**(`node.events`,进节点即给,最该加 `once`)和**选项级 `run`**(`link.run`,点击才给;通常点完跳走、不易重复,可不加守卫,但要具体判断;一次性选项加 `once:true`,引擎据 `_once` 账本消耗掉该选项)。
- [ ] 物品获得是否防重复(`run` 里 `if(!S.inventory.includes('X')) S.inventory.push('X')`,或放在 `once:true` 的 beat/link 里);flag 赋值是否幂等。这两类通常没问题,重点是数值。

### 1.3 删除/重命名节点后,清除所有悬空引用(死链)
- [ ] **先跑 `node core/tooling/graph-audit.mjs world.js`**——它专查死链(`to` 指向不存在的节点)/ 可达性 / 死胡同,P0 退出码非零。结构层的死链优先靠它兜底,再人工补查函数式 `to`。
- [ ] 任何被删除的节点,**全文搜索它的 id**,确认没有任何 `to: '已删除节点'`(或 `to:{map,node}`)残留——包括 link 的 `to`、模块 `actions()` 里动态产出的移动目标。
- [ ] 注意:若审计工具把"已删除节点"直接排除出节点表,可达性检查会显示"全部可达"的**假象**——因为悬空引用指向的目标根本不在表里。要**单独检查"所有 `to` 目标是否都存在于 `maps→nodes` 中"**(graph-audit 已做这一项)。
- [ ] 确认引擎对"未知目标"的处理:核心 `enter` 一个不存在的节点会失败/卡住,**死链 = 玩家卡死**。
- [ ] 删一个结局/分支时,要把**整条管道一起删干净**(触发它的 link、检定节点、`look` 里对应的条件分支、相关 flag),否则留下悬空引用。

### 1.4 属性门槛与防刷守卫的联动(平衡)
- [ ] 加了"防刷属性"守卫后,**必须重新核算属性的真实可得范围**。两者是一对矛盾:防刷会大幅减少属性来源。
- [ ] 算两个极端:**最短主线的属性下限**(完全不探索能拿多少)和**深挖路径的上限**(尽量探索能拿多少)。
- [ ] 用这两个值检验所有门槛:
  - 若"最短主线下限"就超过了高级结局门槛 → 门槛**形同虚设**,没有区分作用。
  - 若"深挖上限"都达不到某门槛 → 该结局/选项**几乎没人能解锁**。
- [ ] 好的门槛应落在"浅通关达不到、深挖能达到"的区间,让探索深度真正影响可选项。

## 第 2 层:状态 / flag / 条件正确性

### 2.1 每个 requires 门槛都必须"可满足且时序正确"（高频坑)
- [ ] 列出所有 link 的 `requires:(S)=>…`(读 flag / 属性 / 物品)。
- [ ] 对每个 `requires` 读到的 flag,**找到它的设置点**(注意:设置点可能在**节点 `events` beat 的 `run`** 里,容易漏!也可能在某个 `link.run` 里),并确认:**在玩家到达这个 `requires` 检查点之前,设置点是否在必经或可达路径上?**
- [ ] 典型 bug:
  - **flag 在被检查之后才设置** → 选项永远是灰的(`showWhenLocked` 则永远灰显)。
  - **flag 设反** → 比如"去A之前"的选项却 `requires:(S)=>S.flags.beenToA`(A 的 flag 在到 A 时才设)。
  - **flag 根本没设** → 正文明明演了"你学会了X",但没有任何 `run` 写 `S.flags.X=true`,导致依赖它的结局/选项锁死。
- [ ] 做一次全局核对:`所有 requires 读到的 flag` 减去 `所有被 run/beat 写入的 flag` = 空集。非空即有"被要求但从未设置"的 flag。

### 2.2 选项不能全部不可选(死局)
- [ ] 对每个节点,检查是否存在"玩家到达时,所有 link 的 `requires` 都不满足"的情况 → 玩家卡死或只能看着灰按钮。模块有"安全网"(无任何可点动作时把隐藏的锁定项灰显出来防空场景死局),但**作者仍应保证每个节点有无条件出口**(一个不带 `requires` 的 link),别依赖安全网。
- [ ] 尤其检查**终局大抉择 / 章节枢纽**:低属性 / 少探索的玩家是否至少有一个可选项?若只剩一个,虽不卡死,但"大抉择只有一个选项"会让玩家觉得前面白玩——至少用 `showWhenLocked:true` + `lockHint` 给灰选项一个解释性的提示。

### 2.3 结局/分支的可达性与条件回顾
- [ ] 确认每个结局**都有至少一条可达路径**(用 `graph-audit.mjs` 的可达性,或实际玩通)。
- [ ] 结局里若有"回顾玩家选择"的条件正文(`look:(S)=> (S.flags.X ? '你曾经…' : '') + …`),确认**每个被读到的 flag/物品都真的有设置点**(某处 `run`/beat 写过),否则该回顾段永不显示(空回顾)。
- [ ] 每个"检定节点 / 不可逆选择"前,最好有**反悔/返回**的退路 link。

## 第 3 层:叙事一致性（这类游戏最易被忽视、最伤沉浸感）

### 3.1 重访节点的文字必须区分"首次 / 重访"（高频坑）
- [ ] 找出所有**多入口 / 可重入**的"地点型"节点(算每个节点的入度 + 是否在环里;`canon-scan` 这类静态扫描可数入边找高入度节点 = 重访候选)。
- [ ] 检查这些节点的 `look` 是否**写死了一次性措辞**(尤其是单字符串 `look` 或 `look` 函数里没分首次分支的),典型:
  - "你**第一次**看到X" / "你**终于**到达" / "一个你**没见过**的人迎上来"
  - NPC **重新自我介绍** / 把玩家当**新人**对话("你也是来画地图的?")——玩家可能已经认识这个 NPC 很久了。
  - **一次性动作**:"你花了两小时**搭建**营地" / "你**初次**踏入"——重访时不该重做。
- [ ] 修法:用 `look` 的**首次/重访**形,引擎按访问计数自动选:`look:{first:'完整初到+介绍+一次性动作', return:'简短「你回到了X」'}`,或 `look:(S,first)=> first ? '首次版' : '重访版'`。**无需手写 flag**——引擎的 `firstTime`(`visits<=1`)就是这个判断。
- [ ] **注意(模型已替你解决这个坑)**:旧格式里人们手写一个 `X_visited` flag 来判首次,极易出错(若误用节点**自己首次进入时就设的** flag,逻辑会反)。Amatlas 的 `first` 参数 / `firstTime` 由核心**访问计数**驱动,且"纯动作不离开就还算首次"(见 lessons-learned),不要再自己造一套首次 flag 与它打架。

### 3.2 时间 / 天数 / 晨昏的一致性
- [ ] 找出所有写死**时间锚点**的节点:"第N天" / "天亮了" / "傍晚" / "日落" / "午夜" / "你醒来"。(时间推进用核心时钟 `S.clock.t` / `api.clock.advance()`,"第几天"的展示是世界/模块层映射,见 module-interface 五。)
- [ ] 区分两类:
  - **章节切换点**(单向推进,如"第一夜结束→天亮",常由 `once:true` 的 beat 推时钟):时间描写合理,**不用动**。
  - **可重入的地点节点**(如营地 `look` 写死"傍晚日落"):玩家可能上午就回来了,或已重访多次,时间描写会**倒流/重复流逝**。把时间描写挪进 `look` 的 `first` 分支,或按 `S.clock.t` 取措辞。
- [ ] 检查多入口汇流点:从不同来路进入同一节点,**时间语境是否冲突**(如有的来路是白天、有的是深夜,但节点 `look` 统一写"午夜")。必要时在 `look` 函数里按 flag/`clock.t` 适配来路,或改用中性措辞。

### 3.3 全局时间线锚点统一
- [ ] 找出贯穿全局的时间锚点(如"主角失踪两年前""某事件发生在X年"),**全文搜索相关数字,确认前后一致**。
- [ ] 典型 bug:同一个人/事件,在 A 处说"两年前",B 处说"三年";日记 A 有年份、日记 B 缺年份导致读者无法对齐。
- [ ] 区分"确指"和"模糊修辞":"对你来说是两年"(确指,要统一)vs"也许是三年,也许是三十年"(角色主观感叹,可保留)。

### 3.4 多入口汇流点的事件时序
- [ ] 对"多条线汇入"的关键剧情节点,画一张**事件先后图**(如:冲突→选边→风暴→风暴后,某角色在风暴中消失)。`graph-audit.mjs` 给出图结构,可据此手画时序。
- [ ] 确认**没有任何路径能违反这个顺序**。典型 bug:风暴后(某角色已死)的节点,却还有 link 能 `to` 回"该角色还活着"的冲突节点。门控用 `requires:(S)=>!S.flags.stormPassed` 之类切断违序回跳。
- [ ] 检查"第几次重排/事件"在不同路径上是否错位(玩家走法不同,可能把"第二次"当"第一次"经历)。

## 第 4 层:内容质量

### 4.1 相邻 / 串联场景的内容重复（高频坑）
- [ ] 凡是 **A→B 且 A、B 都描写同一角色出场 / 同一事件**的,检查 B 是否在**重述** A 已经讲过的内容,而不是**承接**。
- [ ] 典型:A 场景已经"某角色眼睛一亮、从人群走出来、说出主张",B 场景(A 的直接后继)又来一遍"某角色眼睛一亮、从人群走出来"——玩家读到角色出场两遍。
- [ ] 修法:B 的 `look` 用函数形按 flag 分支——`look:(S)=> S.flags.见过A的事件 ? '承接版' : '首次版'`(A 节点用 `events` beat 或 `link.run` 写 `S.flags.见过A的事件=true`),或直接改写 B 的开头为承接。**切勿**在 `look` 字符串里写 `{{if:...}}{{else}}{{endif}}`——呈现器不解析,会原样显示给玩家(见 1.1)。
- [ ] 顺带检查**主题句 / 金句重复**:核心主题句反复出现强化是好的,但若同一句在多个相邻场景重复 4+ 次,会让人觉得啰嗦,适当删减。

### 4.2 空正文 / 内容缺失
- [ ] 找出**正文为空或极短**的场景(只有配图/SVG、没有叙述文字)。区分:
  - **本就是配图过场**(靠选项推进,设计如此)—— 可接受。
  - **应该有内容却空了**(玩家点进来看到空屏)—— 补内容或合并。

### 4.3 NPC / 概念的首次出场要有铺垫
- [ ] 关键 NPC 第一次被命名/对话时,**之前是否有引入**?典型 bug:某场景突然说"族长安排的向导小X带你参观",但小X此前从未出现、族长也没提过要安排向导。
- [ ] 概念同理:正文说"传说中的活桥",但玩家此前从没听说过"活桥"这个传说。改成不预设玩家已知的措辞,或补铺垫。

### 4.4 资源 / 数值变化符合直觉
- [ ] 检查核心资源(金钱/墨水/体力等)的增减是否**符合直觉**:谨慎/正确的选择不应反而扣资源(像惩罚);消耗应可解释(画图耗墨水合理,收拾工具耗墨水不合理)。
- [ ] 术语一致性:同一工具/物品全文用词统一(别一会儿"铅笔"一会儿"碳墨笔"指同一支笔)。

## 第 5 层:可玩性

### 5.1 避免"假选择"(所有选项殊途同归且触发同一剧情)
- [ ] 检查是否有场景的**多个选项全部通向同一个下一场景**,尤其当那个下一场景是**剧情高潮/强制推进点**时——这等于没有选择(玩家点哪个都一样,且都被推进剧情)。
- [ ] 探索类场景尤其要避免:玩家在悠闲探索时,每个选项都"触发剧烈剧情转折",会很突兀。
- [ ] 至少保证关键探索场景有"再逛逛 / 回到安全点"的**退路选项**。

### 5.2 选项措辞要可区分
- [ ] 同一场景里,避免两个选项**措辞几乎相同**但导向不同结果(玩家无法预判区别)。给出能体现差异的措辞。

## 输出规范(给审查者自己的要求)

- **区分确认级别**:每条问题标注"✅已确认(跑通/逐行核对)"或"🟡待确认(静态推断,需运行验证)"。
- **区分严重度**:P0(玩家可见崩坏/卡死)> P1(平衡/时序/语义矛盾)> P2(打磨)。
- **每条给出**:位置(场景名/行号)、问题、**为什么是问题(后果)**、具体修法、验证方式。
- **承认并撤回误报**:一旦发现之前的判断是解析器盲区导致的误报,**立即、明确地撤回**,并说明根因,避免开发者照着假 bug 去改、反而改坏正确逻辑。
- **不要只报问题**:明确列出"已修复/健康、无需改动"的部分,让开发者知道哪些不用动。
- **改动必须在引擎能力范围内**:提的修复方案,必须是引擎实际支持的语法/机制能实现的,否则会把"语义问题"换成"崩坏"。

---

## 一句话总则

> **先把审查工具校准到理解全部运行时机制(否则会制造假 bug);能跑就跑;然后按 引擎正确性 → 状态/flag → 叙事一致性 → 内容 → 可玩性 逐层查;每条结论区分"已确认/待确认",每个修复方案都必须落在引擎实际支持的能力之内。**
