# 跑团游戏设计指南(authoring · 设计原则)

> 配套 `few-shot.md`(给**填好的** schema 例子)。**这份讲"为什么 / 什么时候"**——何时掷骰、角色卡怎么配、资源与时钟怎么平衡、DC 怎么定。
> 引擎只提供机制(`api.dice` + DC + 资源 + 时钟 + 事件总线);**好不好玩取决于这些机制怎么配**。运行时见 `runtime/tabletop.js`,可玩实例见 `examples/tabletop-demo/`。
> 设计依据(均已查证):Citizen Sleeper / Disco Elysium(技能检定 vs 难度替代战斗、时钟/资源驱动叙事)、PbtA(2d6+调整、失败也推进故事)。

## 1. 什么时候用检定(check),什么时候不用

**只在"结果不确定、且成功和失败都有意思"时掷骰。** 检定 = "花资源 + 冒险换一个不确定结果"的动作;不是每个操作都该掷。

- ✅ 撬开卡死的舱门(成功=进门;失败=费时/受伤但人还在)、说服敌意的 NPC、在坍塌前抢修。
- ❌ 走到隔壁房间、捡起明摆着的钥匙、原路返回——**这些用 `exits` 移动**(需要条件就给 `available`),不要做成检定。
- ❌ "没有失败代价"的检定:可无限重试又不扣资源 = 玩家只会刷到成功,等于没掷 → 给它 `cost` 或 `success.flag`(见 §3)。
- ❌ **"通过后还能反复刷"的 cost 检定**:`success` 只写 `text`(没有 `flag` / `set` / `to`)→ 引擎认不出"已通过"、检定按钮不消失 → 玩家可无限重点、每次照扣资源,能把资源刷空、软锁后续(showcase《逝音录》实测)。**给 `success` 一个 `flag` 或 `set`(标记已通过 → 引擎「已通过不重摇」守卫自动隐去检定),或 `to`(通过即前进)**。引擎守卫认 `success.flag` **与** `success.set`(二者都写 `state.flags`、一致对待);唯独纯 `text` 没有"已通过"标记、防不住。

> ⚠️ **掷骰用引擎的 `api.dice`,绝不用 `Math.random()`**:`api.dice` 走种子化 RNG(`services.rng`)→ 同一存档重玩结果一致(可复现)、且产出 `scene.elements` 的 `dice` 图元让 present-svg **动画呈现**(掷动→定格、成败着色,紧张感)。在 `look`/`run` 里写 `Math.random()` = 不可复现(玩家重载存档点数就变)、没有检定 UI、也不会触发骰子动画——是"看起来在掷骰、其实啥机制都没接上"的假检定(showcase round6 实测翻车点)。**要数值鉴定就用 `kind:'encounter'` + `api.dice`,别自己造轮子。**

> 🎲 **骰子外观/换皮**:检定结果由 present-svg 自动据 `dice:'NdS'` 的面数选骰形(d6 等距立方、d20 六边形近似…)+ 自然最大/最小的暴击金光 / 大失败红裂。这些是**素基底,自动生效、零配置**;想换材质 / 配色 / HTML div 真 3D,见 `dice-styles.md`(纯 CSS 换皮,不碰呈现器代码)。

**关键陷阱:别把唯一出路锁在一个"失败=卡死"的检定后面。** 三种正确做法(任选其一):
1. **失败前进(fail forward)**:失败也给后果(扣资源 / 推时钟 / 换一段叙事),故事继续——而不是"再试一次"。
2. **留备选路径**:另一个 `exit`、或换一种技能的检定能绕过去。
3. **失败可重试但有代价**:`cost` 扣资源,资源耗尽前总能过,但耗掉的资源会在后面咬人。

## 2. 角色卡怎么设计(`sheet`)

角色卡 = `{ skills: {技能:调整值}, resources: {资源:数量} }`,经 `createTabletopModule({ sheet })` 注入,**随存档往返**。

**技能 = 小整数调整值,不是原始属性。** 引擎把 `skills[技能]` 直接加到骰子上(`roll + mod >= dc`)。
- **范围克制**:`-1 ~ +3` 足够。`+3` 已是"专家"(能把一次中等检定从五五开拉到大概率过)。
- **3-5 个技能**,域要分得开(体魄 / 感知 / 交涉…),让不同检定能区分角色擅长什么。技能太多 → 每个都用不上几次、失去意义(Citizen Sleeper 只用一把)。
- **只定世界里真会检定的技能**:没有对应 `checks[].skill` 的技能是死字段,别堆。

**资源 = 可消耗的池子**(`状态` / `弹药` / `理智`…),检定的 `cost` 从这里扣。它是**节奏与压力**的来源(见 §3)。

**命名要对齐(对不上=静默失效,装配/审计会 warn)**:
- `checks[].skill` 必须**逐字匹配** `sheet.skills` 的键——`'感知'` 写成 `'perception'`(中/英混用最常见)→ 引擎静默按调整值 **0** 算、检定凭空变难。
- `cost.res` 必须**逐字匹配** `sheet.resources` 的键——拼错 → 静默不扣、且该检定被当"资源不足"**永不可点**。
- **资源名别和文字冒险的状态字段同名(同名两池)**:混合游戏里跑团检定的 `cost` 扣的是**角色卡** `S.sheet.resources.X`,但作者在场景 `link.run` 里天然写**顶层** `S.resources.X`(回血/扣费)或 `S.anxiety`(状态)——**同名两池、永不同步**:场景回血对检定无效、检定只从初值单调干涸、状态条数字与检定实际池背离("反复试也没代价"/"回了血检定还是过不了";showcase 实测**两次翻车**)。**铁律:混合游戏里一种资源只用一个池——场景的回血/扣费/状态条一律读写 `S.sheet.resources.X`(单一真相=角色卡池),别用顶层 `S.resources.X` 或 `S.X`**。**闸**:`graph-audit` 现自动报「同名两池」(world.js 写顶层 `S.resources.X` 且 game.js sheet 声明同名 `resources.X` → P1)。
- **检定后果 `set`/`flag` 写进 `state.flags.*`,别在 `initState` 顶层声明同名字段**:`success:{set:{won:true}}` / `flag:'won'` 落在 `S.flags.won`;而 `initState:{won:false}` 声明的是**顶层** `S.won`(两个不同命名空间)。**flag 不必声明**——读未写的 flag = `undefined`(≈false),引擎已兜底;门控/叙事布尔一律读 `S.flags.X`。`initState` 留给**要参与运算的顶层数值/字符串**(`S.fame`/`S.hp` 这类 `S.x-1` 会用到的,防 NaN)。在 initState 顶层声明 check 会 `set` 的 flag = **幽灵声明**(读不到;且若给了非 false 初值,该初值会丢)。(showcase 武侠局实测:7 个 flag 在 initState 顶层声明、实际走 `S.flags.*`,顶层声明全死。)

## 3. 资源与时钟的平衡(游戏感的核心)

这两个旋钮决定"紧不紧张"。

**资源(resource)——管"能试几次 / 要不要赌"**
- `cost: { res:'状态', amount:1 }` 在**每次尝试**时扣(成败都扣)。资源不足 → 该检定**自动不可点**(`available=false`),天然挡住无限刷。
- **预算法**:资源初值 ≈ 一条主线**必要检定数 + 1~2 缓冲**。给太多 = 失败没代价;给太少 = 一两次失手就卡死。demo 用 `状态:3` 对一条 3-4 检定的短线 = 容错小、每次失手都疼。
- 让玩家在**"重试" vs "省着用"**之间权衡——这是跑团的核心抉择。
- **要恢复资源就显式写回**(休整点 / 补给):在节点 `events[].run` 里 `S.sheet.resources['精力'] = Math.min(最大值, (S.sheet.resources['精力'] || 0) + n)`。注意 `sheet` 在 `state.sheet`(随档),与 `initState` 的世界状态字段是**两套**。**别只在 `look` 文案里写"精力有所恢复"却不动数值**——那是文案对玩家撒谎、资源条纹丝不动(showcase 实测翻车点:文案承诺 vs 代码实现脱节)。
- **改资源的 `run`/`event.run` 一定要 `return` 一句变化文本**(如 `return '补给+3,弹药−2。'`):引擎会把 run 的返回字符串作为一条 beat 显示给玩家;status 状态条**只显当前值、不高亮增减**,所以**不 return 文本 = 玩家根本看不到这次得失**(showcase round12 实测:劫掠节点 event.run 改了声望/弹药/补给三项却没 return → 三项资源静默变化,玩家"不知道发生了什么")。规则:**凡 run 里动了资源/重要状态,就在末尾 return 一句点明增减**——这是"资源得失有反馈"的唯一途径(引擎不替你高亮)。`set:` 只写 `flags`,不显示文本,也不该用来改数值资源(改资源用 run 里直接 `S.x = …`)。

**时钟(clock)——管"时间压力 / 倒计时"**
- `success.clock` / `fail.clock` 推进时钟(`ENG.clock.advance`,**单调只增**,随档)。
- 最有张力的设计:**成功也推时钟**(demo 如此)——"过了,但天又黑了一点"。教会玩家"时间也是资源",而非只罚失败。
- 用时钟当**倒计时门**:`exits` 的 `available:(s)=>s.clock.t < 6` 表示"第 6 回合后这条路封了";或用 `systems`/事件在某回合触发变化。让玩家感到"拖不起"。

**成功 / 失败的后果(`success` / `fail`)**
- 可含 `text`(叙事)、`flag`(置 `state.flags[flag]=true`,门控后续 `exits`/检定)、`clock`(推时钟)、`set`(批量写 flags)。
- **置了 `success.flag` 的检定成功后自动隐去**(不重复掷);没有 flag 的检定可反复掷(直到资源耗尽)——据"是不是一次性事件"选用。
- **`fail` 别留空**:至少给 `text` + 一点 `clock`/资源代价,体现"失败前进"。
- **`label` 只写动作、别手写 DC**:引擎据 `skill`+`dc` 字段**自动**在按钮后拼「(技能·DC N)」——写 `label:'撬开闸门'` + `skill:'力量'` + `dc:7` → 显示「撬开闸门(力量·DC 7)」。别写 `label:'撬开闸门(力量·DC 7)'`(会和 dc 字段重复、改 dc 还会脱同步)。**通过后**那个"已挣得的通路"按钮自动显纯动作「撬开闸门」(不再带 DC,已通过=DC 无意义)。不想自动拼 → `dcHint:false`。

## 4. 串起来:一个紧张的遭遇怎么搭

1. **场景气氛(encounter 必写 `scene`)**:`scene:{ region, mood }` + `audio:{ bgm }` 给氛围(presenter 自动出画/出声,作者不碰素材)。⚠️ **漏写 `scene` = 进入时没画面**——要等点了检定、骰子才把一个空场景顶出来突兀冒灰底(graph-audit 报 P1)。照抄 `examples/tabletop-demo`:每个 encounter 都带 scene。
2. **1-2 个有意义的检定**:不同技能、明确 DC(见 §5)、`cost` 扣资源、成败都有后果。
3. **移动用 `exits`**:用 `available` + 检定置的 `flag` 把"撬开门 → 才能深入"串起来。
4. **时钟收口**:关键检定推时钟,某回合后触发结局/封路,制造倒计时。
5. **检定 → 事件**:要成就/连锁就 `api.on('check', fn)` 订阅(松耦合,不改模块)。

### 4b. 检定分支(v12:`success.to` / `fail.to` —— 失败也把剧情推向新处境)

> **检定的呈现节奏(显示结果 → 继续)**:点检定后**不会立即跳走**——先在**当前 encounter** 显示骰子动画 + 成败结果行 + `success.text`/`fail.text` 后果叙事,再出现一个「继续 →」走到 `success.to`/`fail.to`(CRPG 标准:掷骰 → 看结果 → 推进)。这样检定结果**一定看得见**,无论目的地是 encounter 还是 scene 节点。**作者照常写 `success.to` / `text` 即可**,无需关心这层节奏(引擎自动处理)。

成功/失败各可带 `to:'节点id'`(或 `{map,node}` 跨图):**先结算 text/set/flag/clock,再移动**。这是 fail forward
的正统数据化(Blades in the Dark「worse position」/ Disco Elysium 红检成败皆分支 / Fallen London challenge 双 outcome):

```js
checks: [{ skill:'体魄', dc:8, dice:'2d6',
  success: { text:'翻墙而入。', to:'court' },
  fail:    { text:'摔进沟里,惊动了守卫。', clock:1, to:'ditch' } }]   // 失败=新处境,不是惩罚性原地
```

**后果选型速查(Blades 五型后果 → Amatlas 字段)**:剧情转向(worse position)→ `fail.to` · 推进威胁(complication)→ `clock` ·
持久代价(harm)→ `cost`/`set` 扣资源 · 机会溜走(lost opportunity)→ `flag` + 检定 `available` 门控(失败后此路不通) ·
效果降级(reduced effect)→ `set` 改数值。
- **防磨档**:没有 `to` 的 `fail` 至少改一个会收敛的状态(`clock`/资源)——"失败只显示文字、原地无限重掷"是
  Fallen London 早期被骂最惨的磨档结构;失败无代价就别掷骰(§1)。
- **防瓶颈**(Disco Elysium 教训):若某节点**唯一**前进路径是 `success.to`,给 `fail.to`(哪怕通向更糟的路线)或留一条
  保底出口——graph-audit 会把"只有 success.to、失败原地"的纯检定节点报「无保底出口」,这不是误报,是在帮你。
- 四种组合都合法:都不走(纯状态检定)/ 成走败留(白检式重试,配 `cost` 防刷)/ 双走(红检式分叉)/ 只败走(陷阱)。

(完整可填 schema 见 `few-shot.md` §4;可玩组装见 `examples/tabletop-demo/`。)

## 5. 定 DC 的速查(默认 2d6,均值 7)

`roll + 技能调整 >= dc` 即成功。下表是**裸 2d6** 掷出 ≥N 的概率;角色有 `+m` 技能时,等效目标是 `dc - m`。

| 裸 2d6 ≥ | 概率 | 用作 |
|---|---|---|
| 6 | ~72% | 容易(擅长者几乎必过) |
| 7 | ~58% | 偏易 |
| 8 | ~42% | 中等(无加成略难) |
| 9 | ~28% | 较难 |
| 10 | ~17% | 难 |
| 11+ | ≤8% | 拼运气 / 绝境 |

- **怎么用**:给一个 `+1` 技能的角色定 `dc:8` → 等效掷 ≥7 ≈ 58%,五五开偏顺;定 `dc:10` → 等效 ≥9 ≈ 28%,明显冒险。
- **想"高手稳过、生手够呛"**:别去调 DC,**调角色的 `skill`**——同一 DC 下 `+3` 和 `0` 的成功率天差地别,这正是技能存在的意义。
- **换骰制就要重定 DC**:`dice:'1d20'` 是均匀分布、方差大;`2d6` 集中在 7。换了骰子,上面这张 2d6 的表就不适用,要按新曲线重定。

## 6. 让检定不纯随机:条件骰 / 优劣势 / 暴击大失败 / 道具加值(v16)

真实跑团的检定**会被道具、属性、前序选择改变**——「1d6 几乎不可能,但你之前捡到小刀,这次变 2d6 就有戏」。Amatlas 给四个**可选**手段(全部守确定性 seeded RNG、缺省零行为变化、错形态 fail-loud)。

**① 条件骰 `dice:(S)=>'NdS'`** —— 骰池随状态变(最直接的"道具改难度",= 用户诉求):
```js
{ skill:'力量', dc:9,
  dice: function (S) { return S.flags.hasKnife ? '2d6' : '1d6'; },   // 没刀 1d6 几乎过不了、有刀 2d6 有戏
  success:{ text:'…' }, fail:{ text:'…' } }
```

**② 优势/劣势 `advantage` / `disadvantage`**(布尔 或 `(S)=>bool`;优劣并存抵消=原骰):
```js
{ skill:'感知', dc:8, dice:'2d6',
  advantage: function (S) { return !!S.flags.hasTorch; },   // 有火把→暗处检定优势
  success:{ text:'…' }, fail:{ text:'…' } }
```
**机制 = 多掷一颗骰、保留最高/最低 N 颗**(N=原骰数):`2d6`+优势 → 掷 **3d6 留最高 2**;`1d20`+优势 → 掷 **2d20 留最高 1**(=**D&D 5e 优势**);劣势留最低。这正是 **Mongoose Traveller 2E 的 Boon(3d6kh2)/ D&D 5e(2d20kh1)** 的统一形式——骰子层面取舍、有真实官方系统背书。它**不等价于固定 +N**(2d6 下 3d6kh2 在中段 DC 增益最大、极端 DC 增益小,非线性)。
优势是**二元开关**(有/无)、不堆数值——适合"做对了准备 / 创造了有利局面"。引擎在掷骰时把 `(优势)`/`(劣势)` 标进检定行尾部(`2d6(11)+1 = 12 ≥ DC 7 (优势)`)→ **别在道具拾取标签里剧透机制**:写「速降钢缆(检定时有优势)」破沉浸(showcase 实测翻车点);写「一捆速降钢缆,金属扣冰冷」就好——优势到掷骰时自然显现。道具靠**叙事**描述它是什么,不靠标签标它给什么加成。

**③ 暴击 / 大失败叙事 `crit` / `fumble`**(自然最大且成功→crit、自然最小且失败→fumble;形态同 success/fail:`text/set/flag/clock/to`):
```js
{ skill:'意志', dc:7, dice:'2d6',
  success:{ text:'你顶住了。', to:'x' },
  crit:   { text:'不仅顶住,还窥见了真相。', flag:'sawTruth', to:'secret' },   // 掷出 12 且成功
  fail:   { text:'你退缩了。', to:'y' },
  fumble: { text:'彻底失控,惊动了它。', clock:2, to:'ambush' } }              // 掷出 2 且失败
```
**不写 crit/fumble 也行**——降级用 success/fail(行为不变)。极值很罕见(2d6 各 1/36),给它专门后果 = 难忘的英雄时刻 / 戏剧逆转。想让 crit 也"完成"该检定(隐去不重掷),在 crit 里也置同一个 `success.flag`。

> **来源诚实**:2d6 自然 12/2 = 暴击/大失败**不是** PbtA/Traveller 官方规则(PbtA 只有 6-/7-9/10+ 三档、12 与 10 同档),是 Amatlas 借鉴 d20 自然 20/1 的设计选择;且**要求极值与成败方向一致才点亮**(掷 12 却因 DC 太高仍失败 = 不算暴击,规避了"骰满却失败也暴击"的逻辑别扭)。完全可选、概率 1/36≈2.8%(与 d20 自然 20 的 5% 同量级)。另:暴击**不自动成功**(仍需 `total≥DC`,符合 D&D 5e 技能检定 RAW;与 BG3「自然 20 自动成功」那条 house-rule 不同——想要那效果,用 `dc` 函数形降 DC 或在 `crit` 里置 `success.flag`)。

> **暴击/大失败有没有"额外效果"?——由你决定,引擎不强加。** crit/fumble **没有自动机械效果**(不会自动多给资源/翻倍);它们只是把结果**路由到你写的 `crit`/`fumble` 后果对象**。引擎自动负责的只是让它们**有存在感**:present-svg 的**金光(暴击)/红裂(大失败)视觉** + 检定行**文字标「暴击成功」/「大失败」**。要它们真的"有额外效果",就在 `crit`/`fumble` 里写——暴击→额外揭示 / 捷径(`to`)/ 省下或返还资源(`set`);大失败→并发症(`clock` 推时钟)/ 额外代价(`set` 扣别的)/ 坏分支(`to:'被吞'`)。**不写就和普通成/败一样**,极值只剩视觉、浪费了这个戏剧点。

**④ 道具/状态临时加值 `bonus:(S)=>number`**(在固定 skill 调整之外;检定行显合并调整值):
```js
{ skill:'交涉', dc:9, dice:'2d6',
  bonus: function (S) { return S.flags.hasToken ? 2 : 0; },   // 信物 +2
  success:{ text:'…' }, fail:{ text:'…' } }
```

**还有两个其实早就支持、只是少用**:
- **道具降 DC**:`dc: function (S) { return S.flags.hasCrowbar ? 12 : 18; }`(dc 一直可为函数)——"调节器"式,比硬门更顺滑。
- **白色检定(失败可重试)**:给检定 `available:(S)=>bool` 门控,失败后条件仍满足就能再试(配 `cost` 防刷)。

**⑤ 取高/取低 K 颗 `dice:'NdSkhK'` / `'NdSklK'`**(D&D 属性生成 / Blades 取最高):掷 N 颗、保留最高(kh)/最低(kl)K 颗求和。`'4d6kh3'`=经典属性骰。**与 ② 优势的关系**:两者是同一"保留取舍"数学的两个入口——② `advantage` 是**随状态触发的语义开关**(自动把 NdS 变 (N+1)dS 留 N,如 2d6→`3d6kh2`);⑤ kh/kl 是**作者写死的静态骰型**(如 `4d6kh3`)。想要"做了准备就更稳"的动态优势 → 用 `advantage`;想要固定属性骰/骰池 → 用 kh/kl。可组合:`dice:(S)=>S.flags.lucky?'3d6kh2':'2d6'`。(注:优势对 2d6 产生的是 `3d6kh2`〔留两颗〕,**不是** `2d6kh1`〔掷两颗留一颗〕——两者不等价。)

**⑥ 部分成功 `partial`(+ `partialBand`)= 失败侧的 fail-forward 补偿**(参照 **FATE 的「succeed at a serious cost」**——差一点的失败给一个**带代价的出路**,而非纯卡死):`total` **没到 DC**、但落在 `[dc-band, dc)` 且非大失败 → 走 `partial` 后果(形态同 success/fail),检定行标「部分成功」。带宽 `partialBand` 缺省 2(可设 3)。**opt-in**:不写就没这档(只有普通成/败)。
> **诚实:这是失败侧、不是 PbtA 7-9。** Amatlas 是 `total≥DC` 的二元判定;partial 触发时 `total<DC`(机械上属失败档)→ 引擎据此**播失败音效、检定行显 `<DC`**,这与"带代价的出路"框架是自洽的(你**没够到**,但争取到一个有代价的结果)。它对应 **FATE 的代价成功**,**不是** PbtA 7-9 / Blades 4-5(那两个是**越过成功线**的"成功带代价",在成功侧)。所以 `partial.text` 写成"勉强/带代价地办到",别写成"轻松成功";想让 `on:'check'` 成就在 partial 触发时,别判 `ev.ok`(partial 时为 false)。
```js
{ skill:'交涉', dc:10, dice:'2d6', partialBand:3,
  success:{ text:'他完全信了。', to:'ally' },                                        // total≥10
  partial:{ text:'他将信将疑——肯帮,但要你先证明自己。', flag:'owesProof', to:'trial' }, // total 7-9:没到 DC、但够近 → 带代价的出路
  fail:{ text:'他轰你出门。', to:'street' } }                                        // total≤6
```

**⑦ 骰池随资源缩减(Citizen Sleeper 风)= M1 的应用**:`dice:(S)=>Math.max(1, S.sheet.resources.体力) + 'd6'`——体力越低骰越少。**无需新字段**,条件骰函数即可。

**⑧ Position/Effect(Blades 风,掷骰前定赌注)= 用既有字段表达**:不同 `fail.to` 表达"位置"(risky→退一步 / desperate→重伤),`success.set` / `bonus` 表达"效果级"。开放词汇、零新字段。

**⑨ 骰子显示的是最终鉴定值(不是裸骰点)**:`2d6+2` 掷出 8 时,骰子动画显示 **10**(= 拿去比 DC 的那个数);检定行另给完整拆解 `2d6(8)+2 = 10 ≥ DC 9`(裸骰 8、加值都在文字括号里)。**你不用做任何事**——引擎自动的,提到只为让你知道:玩家一眼看到的数字就是结果数,所以 DC 定得让人"差一点"时张力最足(见 §5)。(与博德之门**不同款**但同样合理:BG3 骰面显**自然骰**、修正值飞入叠加;Amatlas 骰面直接显**总值**、自然骰在检定行文字里——两种都行,Amatlas 选了"一眼即结果"。)

**让成败真的有回响(别让失败只是一句当下文字)**:检定 `success`/`fail`/`crit`/`fumble` 里 `set`/`flag` 置的状态,要在**后续**被读到才有意义——读在某个 `look:(S)=>…` 的分支、某个 `requires`/`available`、结局节点的文本变体、或 `game.js` 成就的 `when` 里。**只 set 从不读 = 玩家的成功/失败没留下痕迹**(showcase 实测:一局里 8 个"受伤/顺利"flag 设了却没接进任何后续 → 玩家觉得中段成败无所谓)。至少让**关键检定**的结果影响后续一段叙事 / 一个出口 / 一个结局变体 / 一个成就。`graph-audit` **不报**"set 了从不读"(可能留作将来用),所以这条回路得你自己接上。

**设计取舍(钥匙 vs 调节器)**:`requires` 硬门(钥匙,完全锁)给"探索/找钥匙"的意义,但用多了易死局;降 DC / 加骰 / 优势(调节器)给"角色构建/凑修正值"的意义、更安全。黄金平衡 = **少量钥匙 + 多数调节**(LucasArts / CRPG 共识)。把这些手段用起来,检定就从"走过场的随机"变成"你的准备和选择真的有回报"。

> **资源预算别造死局**(showcase 实测翻车点):若某节点的**唯一前进路径**是一个耗资源的检定,而失败会绕回该节点、且沿途**无资源恢复**,资源耗尽后就**软锁**(检定灰显、出不去)。要么 fail forward 到别处(`fail.to` 不绕回)、要么留一条无消耗的保底出口、要么在循环里放个回复点(见 §3 资源恢复写法)。`graph-audit` 看不到资源消耗(运行时语义),这个得靠你自己把预算算够。
