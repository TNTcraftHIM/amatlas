# 建造纪律(building discipline)

适用:在引擎里**做游戏 / 扩展引擎**(新模块·呈现器·插件)时的行为底线——所有任务都遵守,弱模型尤其每次都要被提醒。四原则改编自 Karpathy 的 LLM-coding 准则,落到 Amatlas 的数据驱动回合制语境。

1. **不假设,先校准**:动手前先**说出**你对需求的理解与关键假设(别默默开工)。不确定该用现有模块还是新建 → 看 SKILL 类型路由 / `references/plugin-development.md` 判断。假设能选合理默认就**说出假设 + 选默认 + 继续**,只有真无法默认时才打断人。
2. **不过度复杂,从最小可工作版本起**:新模块/插件**复制最接近的现有示例 → 改名 → 逐步替换**(见 `references/plugin-development.md`),别从空白设计复杂架构;先让最简版跑通再迭代。world.js 能用 3 个节点表达的别写 10 个。
3. **增量验证,写一个测一个**:每完成一个子步就跑验证(`node test/run.cjs` / build / `node --check`),别写完全部再测。失败时**先读懂为什么失败**再改,不盲目重试(呼应 `auditing-principles`:先校准工具再下结论)。
4. **目标驱动**:动手前写一两句**给自己的计划**(目标 / 步骤 / 每步"做完长什么样"),随即执行;每步成功标准要**可验证**("view() 返回含 count 的 View",而非"做完就行")。

> **与 `craft-and-autonomy` 自主协议一致**:这里的"计划/假设"是**自导航**——一两句、随即动手,**不是请示、不替代动手**;假设是"说出来 + 选默认继续",不是停下等批准。"落盘纪律"(子步小、每步刷新 `PROGRESS`、决策落盘)同样在 `craft-and-autonomy`——本规则管"怎么把事做对",落盘纪律管"做的状态别丢"。
