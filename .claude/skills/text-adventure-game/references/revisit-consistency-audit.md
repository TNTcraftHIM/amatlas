# Hub 场景重访一致性审计方法论

## 问题本质

分支叙事游戏中，某些"枢纽场景"（Hub Scene）被多条路径指向。
玩家会反复经过这些场景，但正文写死了"第一次到达"的措辞，
导致重访时出戏（NPC 被重新介绍、时间倒流、环境描写重复）。

## 系统性发现方法

### Step 1: 统计 in-degree

```bash
# 统计每个节点被 link 的 to 引用的次数(模块化的链接字段是 to，不是 next)
grep -oP "to:\s*['\x60]([a-z_]+)['\x60]" world.js | \
  sed "s/to:\s*['\x60]//;s/['\x60]//" | \
  sort | uniq -c | sort -rn | head -20
```

也可直接用核心审计工具一次性给出可达性/入度视角:`node core/tooling/graph-audit.mjs world.js`。
入度 >5 的节点 = Hub 场景,需要逐一检查。

### Step 2: 检查是否有首次/重访分支

对每个 Hub 节点,看它的 `look` 是否区分首次与重访。
若 `look` 是单一字符串(对首次和重访都返回同一段) → 该节点需要拆出首次/重访分支:
改成 `look:{first:'…', return:'…'}` 或 `look:(S,first)=> first ? '…' : '…'`。
(模块化由引擎按访问计数自动判定 first;呈现器不解析 `{{if}}` 标记——写了也原样显示。)

### Step 3: 检查时间描写

搜索"日落""傍晚""两个小时""清晨"等时间词。
如果在条件分支之外 → 重访时会时间倒流。

### Step 4: 检查 NPC 介绍

搜索 NPC 名字（如"卡伊""族长"）+ 介绍性措辞（"叫""年轻人""安排"）。
如果在条件分支之外 → 重访时 NPC 会被重新介绍。

### Step 5: 检查 flag 设置

若首次/重访分支是靠某个 flag(而非引擎自带的访问计数)判定的,确认那个 flag 在所有进入该节点的链接 `run` 里都被设置:
```bash
grep -A3 "to:\s*['\x60]camp_setup['\x60]" world.js | grep 'flags'
```
(注:Hub 的首次/重访本身**优先用引擎访问计数**——`look:(S,first)=>…` 或 `look:{first,return}`,无需手动 flag。下面 flag 相关的注意事项,只在你额外用 flag 控制某段条件文本时才适用。)

## 修复模板

```javascript
// 在 world.js 的节点定义中(maps.<map>.nodes.<node>)
// 首选:用引擎自带的访问计数判定首次/重访(无需手动 flag)
{
  kind: 'scene',
  name: '营地',
  // 对象形:首次 / 重访两段
  look: {
    first:  '首次版正文。完整描写、有时间、有 NPC 介绍。',
    return: '重访版正文。简短、无时间描写、无 NPC 重新介绍。'
  },
  // 选项 = links;移动用 to,无需手动设 visited(first/return 由引擎按访问计数自动切换)
  links: [
    { label: '选项A', to: 'next_scene' },
    { label: '选项B', to: 'other_scene' }
  ]
}

// 函数形:首次/重访之外还要叠状态分支时用 (S, first)
look: function (S, first) {
  if (!first) return '重访版正文。简短、无时间描写、无 NPC 重新介绍。';
  return '首次版正文。完整描写、有时间、有 NPC 介绍。';
}
```

> 模块化呈现器**不解析** `{{if}}/{{else}}/{{endif}}` 标记——把首次/重访逻辑写进 `look`(对象的 `first/return` 或函数的 `first` 形参),而不是写进正文字符串里。引擎按访问计数(节点第一次进入 `first===true`,之后 `false`)自动切换,无需在每个选项里手动设 visited flag。

| 场景 | 入度 | 问题 | 修复 |
|------|------|------|------|
| camp_setup | 9 | "搭两小时遮棚+日落北偏西"每次都播 | 首次播完整版，重访播"遮棚还在" |
| deep_forest | 9 | "跋涉两小时+发现倒木+R.H.刻痕"每次都播 | 首次播完整版，重访播"你知道规律了" |

| village_entrance | 4 | "菌门初见+陌生女人迎接"每次都播 | 首次播完整版，重访播"菌门认出你" |

## 首次/重访判定注意事项

**首选用引擎访问计数**(`look:{first,return}` 或 `look:(S,first)=>…`):引擎在节点第一次进入时给 `first===true`、之后 `false`,这是判首次/重访的标准做法,**不需要自己设任何 flag**。它天然规避了下面靠手动 flag 时的三类经典坑。

只有当你**额外**用一个 flag 控制某段条件文本(例:看过笔记后才显示的一句旁白)时,以下手动-flag 的注意事项才适用——它们记录的是 bug 背后的原理:

1. **不要用本节点自设、且在离开时才置位的 flag 判断是否首次**
   - 例:village_entrance 自设 `arrived_village`,但在离开时(链接 `run` 里)才设置
   - 于是进入时检查 `arrived_village` 永远是 false → 首次分支永远触发
   - 要么直接用引擎 `first` 形参,要么引用其他节点设置的 flag(如 `met_chief`,在 meet_chief 节点的 `run` 里设置)

2. **若靠 flag 判定,该 flag 必须在所有离开本节点的链接 `run` 里设置**
   - 如果有 3 条 link 但只在 1 条的 `run` 里设了 visited → 走另外 2 条离开后再回来仍是首次版
   - 最安全的做法:每条 link 的 `run` 里都设 `S.flags.SCENE_visited = true`
   - (用引擎 `first` 形参则无此负担——这正是首选它的原因)

3. **不要把 visited 设在进入即执行的 `events`(beat)里**
   - 节点的 `events`(进入 beat)在每次进入时评估(带 `once:true` 时只生效一次)
   - 但玩家"到达"≠"离开"——若想靠 flag 区分,flag 应在离开时(链接 `run`)设置
   - 这样如果玩家在选项页刷新(未离开),仍能看到正确的首次/重访文本
   - 同样地:引擎 `first` 形参由访问计数驱动,纯动作(不移动的 link)不会被算作"再次进入",不会误把首次翻成重访,无需操心这一步

## 时间描写在 first / return 两段中的分配

下表的"首次分支""重访分支"对应 `look` 的 `first` 段与 `return` 段(或函数里 `first` 为真/为假的分支):

| 描述类型 | first(首次) | return(重访) |
|----------|------|------|
| 绝对时间("日落""傍晚") | ✅ 放入 first 段 | ❌ 不放(会时间倒流) |
| 相对时间("大约两小时") | ✅ 放入 first 段 | ❌ 不放(暗示重新走) |
| 环境描写("潮水在盐渍线处停住") | ✅ 放入 | ✅ 可保留(物理事实不变) |
| NPC 行为("卡伊带你参观") | ✅ 放入 first 段 | ❌ 改为"卡伊不在"或省略 |
| 物资状态("淡水还剩一天半") | ❌ 不放(首次没检查过) | ✅ 放入 return 段 |

## 在 look 里写条件分支的注意事项

Hub 场景的首次/重访是 `look` 区分两段(`{first,return}` 或 `(S,first)=>…`)最常见的使用场景。
模块化呈现器**不解析任何 `{{}}` 标记**——所有分支都是普通 JS,写在 `look` 函数体里。因此没有旧引擎的语法限制,你直接用语言本身即可:
- 多分支并列 → 普通 `if/else if`(不用 `{{elif}}`)
- 按值相等切换 → 直接 `if (S.flags.x === 'foo')`(不用只靠布尔 flag 迂回)
- 嵌套条件 → 普通嵌套 `if`(无嵌套深度限制)

只要 `look` 最终返回一个字符串即可;首次/重访本身由引擎访问计数经 `first` 形参传入,无需在字符串里嵌标记。
