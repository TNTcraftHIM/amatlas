# 我们的引擎:设计哲学与底层逻辑

> 这是引擎最上层的设计文档,回答"我们的引擎到底是什么、凭什么这样设计"。
> 它不是功能清单。"无限循环不出现""玩家知道自己在哪""重访有新内容"——这些都是**结果/副产品**,
> 不是设计初衷。设计初衷是一个**自洽的底层模型**,好行为从模型里**自然涌现**,而不是靠一条条补丁。

## 〇、核心命题:状态驱动的世界模型(state-driven world model)
**我们的引擎是"一个世界模型之上的状态机"。世界状态是唯一真相源。玩家不是"导航到一段存好的内容",
而是"做出动作 → 改变世界状态";每一屏画面都是当前状态的纯函数。**

一句话三件事:
- **状态是真相**(State is truth):游戏 = 一个世界状态对象。位置、时间/阶段、flag、物品、数值、关系、记忆,全在这里。
- **内容是状态的视图**(Content = f(state)):每一屏不是"取出第 N 个节点",而是"按当前状态渲染出来"。
- **动作是状态转移**(Action = transition):选项不是"走一条边到另一节点",而是"改变状态",然后引擎重新渲染。

好行为是这个模型的**推论**:见 §四。

## 一、根因:为什么"节点图"模型必然产生那些症状
**节点图架构**(原生 Twine / 多数 CYOA / 我们现在的引擎):源码与玩法都围绕"原子节点 + 连接"。
**进度 = 你在图里的位置**;一个"场景"把三件事揉成一团——**内容**(写了什么)+**进度态**(你到哪了)+**地点**(你在哪)。
移动 = 沿一条边走。**这个模型里压根没有"什么都没变"这个概念**:每条边永远等价、永远可走。
于是——无限来回、进村总播"第三天"、重访赛琳没新内容——**都不是 bug,而是这个模型允许的正常状态**。
在节点图上靠"加 if 判断来路""加 once 标记"去堵,是在**对抗模型**,所以越补越乱。

**世界模型架构**(Inform 一脉)恰相反:状态被表达成关系/属性(玩家在某处、携带某物、知道某事、第几天),
行为与内容由**规则**从状态推导;"你在哪"只是一个状态变量;重访渲染的是**当前**状态。这里没有"节点",
只有不断演化的世界 + 你对它的观察。

## 二、成熟引擎的底层逻辑(已查证,取其神)
- **Inform 7(世界模型标杆)**:世界 = 实体 + 关系 + 属性("玩家 在 厨房");**规则**读状态、改状态;
  内容是状态的函数(房间描述按 visited/状态变化;首次≠重访是内置的)。**scene 用"begins when 条件"触发**,
  能移动玩家、推进时间——事件是"挂在世界上、被状态触发"的,不是写死的节点链。房间 + 对称出口构成空间骨架。
- **Ink(叙事流标杆)**:故事 = 带记忆的内容流 + 全局变量。**访问计数是整数**(内置首次/重访);
  **选项默认一次性 `*`(选过即消耗)**、`+` 才粘性。本质是"内容有'看过'记忆,默认向前,不回头空转"。
**取舍**:我们做的是叙事/选择向的游戏(有数值、flag、物品、关系),不需要 Inform 那么重的物理模拟;
但要拿走它的**灵魂**——状态是真相、内容是状态的函数、事件被状态触发;再叠加 Ink 的**记忆与默认向前**。

## 三、我们引擎的模型(综合,可实现的抽象)
1. **WORLD(作者写,基本静态)**:世界的结构。
   - `places`:地点 + 相邻**连接**(空间骨架,约定对称双向)。非空间游戏里"地点"可退化为"情境",骨架原则不变。
   - `entities / npcs / items`:有自己状态的对象(NPC 的态度、物品的语义)。
2. **STATE(运行时,唯一真相)**:`{ location, time/phase, flags, inventory, stats, relationships, memory{seen,done} }`。
   整局游戏 = 这个对象的演化史。
3. **render(STATE) → 视图**:由状态**推导**出——当前地点描述(按时间/记忆/flag 变体)、可用动作
   (移动=从连接生成 + 状态门控的 beats)、当前触发的事件(状态触发、可消耗)。
4. **applyAction(action) → 状态转移**:移动改 `location`;重要动作推进 `time/phase`;其余改 flag/物品/数值/关系。
   转移后重新 render。
5. **引擎保证的不变量(invariants)**——好行为从这里涌现:
   - **叙事时间单调**:phase/day 只前进、不回退 → 时间性插页天然只播一次。
   - **内容恒为当前状态的函数** → 不会有"陈旧节点被原样重播"。
   - **移动只沿真实相邻** → 不会有莫名其妙的传送环。
   - **记忆(seen/done)默认让内容不重复** → 重访自动呈现"之后"的样子。

## 四、症状如何"由构造消失"(不是补丁)
- **"第三天"反复播**:day 是状态;day→3 的插页绑在**时间推进这个转移**上,day 单调 → 只播一次。(不是"加 once flag"。)
- **几个选项无限来回**:模型里没有"环"——只有状态演化。移动只改 `location`;内容是状态的函数;
  该一次的动作是"翻转状态的转移",其结果无法无意义复发。"反复走同一条内容边"在这个模型里**不存在**,因为内容不是边。
- **重访赛琳没新内容**:NPC/地点内容 = f(状态, 记忆)。赛琳的话说尽了(记忆里记下了),渲染出的就是
  "赛琳只是望着海,没有再开口",而不是把旧节点再播一遍。

## 五、那些"原语"是模型的**实现**,不是模型本身
访问计数、`once` 选项、`look` 的首次/重访分支(`look:{first,return}` 或 `look:(S,first)=>…`) —— 是用来实现"记忆"和"内容=状态函数"的**机制**,正确且必要,
但它们服务于上面的模型。**先有模型(设计初衷),才有这些机制(实现手段)。** 别再把"消灭循环"当目标来逐个堵。

## 六、流程:先世界,后填充
1. 故事大纲 → 2. **世界**(地图:地点+连接+region;实体:NPC/物品+初始状态)→ 3. 把 **beats(状态门控的内容 + `run` 状态转移)**布到世界上
→ 4. 写内容时一律按"当前状态"来想(首次/重访/时间/关系)。
**作者写的是"世界 + 一组由状态触发的 beats",不是"场景→场景的边"。** new-game 命令按此顺序引导。

## 七、工具(守住模型)
退化环 / 可重入但内容不感知状态 的检测:用来抓"违反了模型"的地方(旧图遗留、或过渡期)。在状态驱动模型下这类问题本不该出现,
检测是防回潮的护栏。

## 八、通用 vs 游戏专属
- **通用(进引擎)**:状态驱动内核、render(state)/applyAction、世界结构抽象、记忆与单调时间、退化环检测、地图优先流程。
- **游戏专属(留游戏)**:具体地图/实体、beats 内容、剧情。

## 九、现有游戏怎么办
现游戏是纯节点图,当**练手作**。先把引擎按本设计做扎实(确立标准),**再**回头按新模型 retrofit;能用"岛屿时空混乱"圆的就先圆。
