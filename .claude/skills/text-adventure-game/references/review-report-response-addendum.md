# 用户行为模式（追加）

## 逐项追踪修复状态

用户会逐一追问每项报告问题是否已修复（"这点修了吗？""墨水经济调整了吗？"）。
即使你已经在前面的对话中修过，用户仍会确认。

**正确做法：** 直接 grep/读取代码验证并展示证据，而不是说"已经修过了"。
用户信任代码证据，不信任 agent 的口头声明。

**示例回应：**
```
已经有了。逐个确认：
| 结局 | 条件 | 状态 |
|------|------|------|
| coexist | flag: learned_flow | ✅ |
| truth | understanding: 5 | ✅ |
...
```

## 精确修复指令

用户会给出具体的年份、替换文本、平衡数值。
如"约翰 1821、玛丽 1838、弗朗茨 1855、赛琳娜 1870、托马斯 1884、雷纳 1895"。

**严格按照用户给的值修改，不要自行调整。**

## echo_chamber 年份对齐技巧

当用户要求让六代制图师年份与代际顺序一致时：
1. 建立目标序列：按代际顺序分配年份，间隔 13-17 年
2. 先改最晚的（雷纳），倒推前面的
3. 如果某代没有留言（如弗朗茨），需要新增一行墙面描写
4. 修改后验证墙上顺序是否严格按年份递增
5. 注意"第三行/第四行/第五行"的序号也要同步调整

## 分阶段结构性审计

用户可能发送多阶段审计报告（阶段0、阶段1、阶段2...），每阶段覆盖游戏的不同部分。
报告格式通常是：读完场景列表 → 问题清单（🔴🟠🟡🟢）→ 待修清单。

**关键行为模式：**
1. 用户会在每阶段开头声明"读完场景:X, Y, Z"——确认覆盖范围
2. 问题按严重程度标记颜色（🔴致命/🟠中等/🟡低/🟢无问题）
3. 每阶段末尾有"待修清单"——直接按清单修复
4. 用户会在后续阶段引用前几阶段的发现（"佐证（见阶段2）"）
5. 用户会记录"我的观察"——这些是给 agent 的元反馈，不是待修项

**处理流程：**
1. 收到阶段 N 报告 → 先验证阶段 N-1 的修复是否仍有效（expanded 覆盖陷阱）
2. 逐项 grep 验证报告中的断言
3. 按待修清单修复
4. 修复后发送文件 + 修复摘要
5. 在 PROGRESS.md 中记录本阶段修复内容
