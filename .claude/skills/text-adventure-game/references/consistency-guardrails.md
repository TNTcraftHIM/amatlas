# 一致性防护:已知失败模式 + 对策

> 来源:ACL 2026 "Lost in Stories" (ConStory-Bench,Microsoft+SUTD)、NousResearch autonovel、
> Rettberg & Wigers 2025、行业"personality homogenization"现象、以及本项目的多轮审计实战。
> 每个防护都对应一个**已被定量验证或反复出现**的真实失败模式。
> 
> 本文件同时服务于两个角色:
> - **写作时的自查参考**:写到哪就核对对应段
> - **reviewer 子代理的检查清单**:narrative-reviewer 循环时按此逐项核对

---

## 一、中段塌陷协议(最重要的结构性防护)

### 为什么
ConStory-Bench 实测:一致性错误**在叙事中段达到峰值**。
- 开头(场景 1-15):世界在建立,注意力高,结构约束强(要设定一切)
- **中段(场景 16-40):最自由 + 最有负担 = 最危险**(无开头/结尾的结构纪律,同时要追踪最多状态)
- 结尾(场景 41+):收束提供了结构,紧迫感提升注意力

### 对策:分段变频审查

> 注:ConStory 原文说"tend to appear around the middle"(针对 8K-10K 字故事),下面的场景号是针对 40-60 场景游戏的工程映射(约中间 40%)——**不是论文直接给的数字**。游戏更短或更长时按比例调整。

| 阶段 | 场景范围(以60场景为例) | review 间隔 | 额外检查 |
|------|---------|------------|---------|
| 建立期 | 1-15 | 每 15 场景 | 标准 reviewer |
| **危险期** | **16-40** | **每 10 场景** | **强制 canon 全量交叉核对 + 角色声音核对 + 分支收束检查** |
| 收束期 | 41+ | 每 15 场景 | 标准 reviewer + 结局回顾核对 |

"危险期"的 reviewer 除了标准检查,**必须额外做**:
1. 把 canon.md 读一遍,逐个核对新场景的时间/角色/世界设定/事实数字
2. 对比每个 NPC 在新批次的对话 vs canon 里的声音速写(见"反同质化")
3. 检查从第 16 场景到当前,是否存在一个 bottleneck/汇流点(见"分支收束")

## 二、ConStory 五维一致性检查表(每次 reviewer 循环必过)

这是 ACL 2026 论文 ConStory-Bench 定义的五类一致性错误。reviewer 每次循环按此结构逐项检查这批新场景,**不是脑扫而是逐条过**。

### 1. 角色一致性(Characterization)
- [ ] **记忆矛盾**:角色 A 在场景 X 声称不知道某事,但在场景 Y 又提起它(或反之)?
- [ ] **知识冲突**:角色知道它不应该知道的信息(没有路径让它获知)?
- [ ] **能力波动**:角色在 A 场景展示的技能/能力,在 B 场景突然不会了(无剧情理由)?
- [ ] **遗忘能力**:角色曾做过的重要事,后续完全不提、不影响行为?

### 2. 事实细节(Factual Detail)—— ConStory 发现:最高频错误类型
- [ ] **外貌不符**:角色外貌描述是否前后一致(发色/身高/疤痕/穿着)?
- [ ] **名称混淆**:人名/地名/物品名是否全程统一(没有同义替换造成歧义)?
- [ ] **数量错误**:数字(距离/时间/数量/年龄)是否和 canon 一致?"两年前"还是"三年前"?瓶中墨水还剩多少盎司?

### 3. 叙事风格(Narrative Style)
- [ ] **视角漂移**:全程第二人称("你……"),有没有突然滑到第三人称或第一人称?
- [ ] **语气断裂**:场景内部的语气是否一致?同一场景从"紧张对峙"突然变成"轻松解说"?
- [ ] **风格断裂**:某场景的文风突然和全局不一样(比如突然学术化或过于口语)?

### 4. 时间线与情节(Timeline & Plot)—— ConStory 发现:第二高频
- [ ] **时间矛盾**:不同场景对同一事件说的时间不一样?
- [ ] **持续时间错误**:"走了两个小时"但两个场景后才"过了十分钟"?
- [ ] **因果违反**:结果出现在原因之前?角色对尚未发生的事做出反应?
- [ ] **被遗弃的情节线**:某条铺垫/伏笔/设置,到目前为止再也没有回应?这不一定是 bug(可能在后面),但必须追踪(见 canon.md 的"伏笔追踪")。

### 5. 世界设定(World-building & Setting)
- [ ] **规则违反**:世界有明确规则(魔法系统/技术限制/岛屿的重排规律),有没有被违反?
- [ ] **社会规范冲突**:文化/社会行为是否一致(如"原住民从不用铁器"但后来出现了铁器)?
- [ ] **地理矛盾**:地点描述一致吗?"海滩在东边"但另一场景说"从海滩向西走到森林"?

## 三、反人格同质化(Anti-Homogenization)

### 为什么
行业观测:"personality homogenization"——所有 NPC 随写作推进逐渐变温和、无冲突、赞同主角。这不是一开始就有的,是**context 填满、模型渐忘角色定义后的退化**。

### 对策

**摩擦配额(Friction Quota)**:
每次 reviewer 循环检查(跳过纯环境探索、无 NPC 对话的批次):新这批场景里,**至少有一个 NPC 做了以下之一**:
- 拒绝了主角的请求
- 误解了主角的意图
- 提出了自己的盘算/条件
- 说谎或隐瞒
- 因自己的事分心/不配合
- 为错误的理由同意

如果所有 NPC 在这批里全部无摩擦同意 → 标记 `[HOMOGENIZATION]`,必须在至少一个 NPC 互动里加入摩擦。

**角色声音漂移检测**:
reviewer 对每个在新批次里有对话的 NPC:
- 读 canon.md 里它的声音速写(词汇/句式/口头禅/绝不会说的话)
- 新对话是否符合?如果一个"从不直接回答"的角色突然直白回答了、一个"短句+金属比喻"的角色突然说了长段诗意隐喻 → 标记 `[VOICE-DRIFT]`。

## 四、信息经济控制(Anti-"立刻揭示一切")

### 为什么
Rettberg & Wigers 2025:AI 故事的通病之一是**信息经济缺失**——立刻揭示一切,无法维持悬念或延迟回报。

### 适用性
**强烈推荐**:悬疑/解谜/有重大揭示的游戏(世界真相、角色秘密、阴谋)。
**可选**:纯情感/探索驱动的游戏(若无需隐藏的信息,跳过此节)。

### 对策

在 canon.md 里维护一个"**信息释放清单**":

```markdown
## 信息释放控制
| 信息 | 何时可以揭示 | 条件 | 当前状态 |
|------|------------|------|---------|
| 雷纳还活着(以守护者形态) | Act 3 | 玩家到达岛屿之心 | 未揭示 |
| 岛屿是远古文明的工具 | Act 3 descent_visions | 无条件(必经) | 未揭示 |
| 制图师传统的起源 | Act 3 | 必须先知道"岛屿是活的" | 未揭示 |
```

**规则:**
- 重大信息必须**有前置条件/探索深度要求**才能揭示,不能因为叙述方便就随手说出来。
- 角色不主动提供玩家没有问的信息(真实人类会这样)。
- 铺垫(foreshadowing)和回报(payoff)必须在 canon 里配对追踪。没有回报的铺垫 → reviewer 标记 `[ABANDONED-THREAD]`。

## 五、分支结构收束检查

### 为什么
Branch-and-Bottleneck 结构(Inkle 验证的最佳模式)要求**每 8-12 个场景有一个汇流点**。没有收束的分支会无限增殖,导致:
- 内容量爆炸(写不完)
- 一致性更难维护(每条分支都要单独追踪)
- 玩家难以体验到"选择的后果在汇流时碰撞"的戏剧性

### 对策
每次 reviewer 循环(尤其中段危险期),检查:
- 从上一个 bottleneck 到当前,有多少场景?超过 12 → 需要一个汇流点。
- 当前有几条活跃分支?超过 3 → 需要至少两条在下一个 bottleneck 合并。
- 汇流点的节点是否处理了不同来路的状态差异(在 `look` 函数里按 flag 分流:`look:(S,first)=> S.flags.X ? '…' : '…'`)?

**简易检测**(审计时):
```bash
# 读 world.js,统计每个节点的入边(in-degree),找潜在 bottleneck(高入度=多条路径汇流)
node -e "const w=require('./world.js');const ind={};for(const mapId in w.maps){const m=w.maps[mapId];for(const nid in m.nodes){const edges=[...(m.nodes[nid].exits||[]),...(m.nodes[nid].links||[])];for(const e of edges){if(!e||e.to==null)continue;const k=typeof e.to==='string'?mapId+'/'+e.to:e.to.map+'/'+e.to.node;ind[k]=(ind[k]||0)+1;}}}Object.entries(ind).sort((a,b)=>b[1]-a[1]).slice(0,10).forEach(([k,n])=>console.log('  '+k+': in-degree '+n));"
```
(`node core/tooling/graph-audit.mjs world.js` 也会从核心契约角度报死链/孤儿/死胡同。)

## 六、逐场景量化监控(轻量,防退化不可见)

### 为什么
每场景字数、选项数、感官细节数——在写的时候看不出趋势,等审查时已积累了 30 个薄场景。需要一个轻量的逐场景记录让退化**可见**。

### 对策
**这是 reviewer 的工作,不是写作者的额外负担。** narrative-reviewer 在审查每批场景时,顺手统计并输出指标表(它已经要读全部内容做一致性/slop 检查,统计是附带的)。

写作者只需:看 reviewer 返回的红线标记,修对应场景。然后把本批的总结(多少场景、几个红线、已修)写进 PROGRESS.md。

**红线(reviewer 检查后标记)**:
- 字数 < 该批平均的 60% → 标记 `[THIN]`,扩写后再继续
- 选项 = 0 或 1 (非结局场景)→ 标记 `[FORCED]`,加选项
- 感官 < 2 → 标记 `[FLAT]`,补感官
- surprising_detail = ✗ → 标记 `[GENERIC]`,加场景专属细节

这个表也是 reviewer 的"证据"——它不靠脑扫,靠量化红线。

## 七、Flag 命名空间管理

### 为什么
60 场景的游戏可能有 30-50 个 flag。如果命名随意(`x`/`thing`/`done`),后半段写到某个 requires 时根本想不起这个 flag 是什么意思,或跟另一个 flag 混淆。

### 对策
**命名规范(Phase 1 建立,全程遵守)**:
```
前缀_描述
  met_chief        — 见过族长
  learned_flow     — 学会了流动记录法
  kol_alive        — 科尔还活着(路径dependent)
  ending_transcend — 走了超越结局
  item_获得时场景   — 获得某物品(如 item_crystal_cave)
```
- 前缀按类型:`met_`(见过NPC)、`learned_`(学会技能)、`visited_`(到过地点)、`_alive`(角色状态)、`ending_`(结局)、`item_`(物品相关)。
- **canon.md 的"物品/Flag 语义"表必须同步更新**。
- 审计时检查:有没有语义重叠的 flag(`arrived_village` vs `visited_village`)?有没有设了但从未检查的 flag(dead code)?

## 八、路径前提假设(Path Prerequisite Assumption)—— 分支叙事最经典的 bug

### 为什么
一个场景可以从多条路径到达,但它的内容(选项文字、叙述、NPC 引用)暗含"你走了某条特定路径才到这里"的假设。非主线玩家到了这里,会看到:
- "**回到**村庄"——但他从没去过村庄
- "你想起了**族长说过的话**"——但他没见过族长
- "经历了**风暴**之后的你"——但他走的路径绕过了风暴

这不是 requires 门控能自动拦的(requires 拦的是"链接是否显示",不是"链接文字/正文是否合逻辑")。

### 结构层能查到的部分(graph-audit + 人工扫一遍)
`node core/tooling/graph-audit.mjs world.js` 从核心契约角度报死链/孤儿/死胡同,并能算出每个节点的入边。**找路径前提嫌疑**靠这两步:
- 用上面"分支结构收束检查"里的入度脚本(直接读 world.js)找所有 in-degree ≥ 2 的节点(多条路径可达)
- 对这些节点,人工/reviewer 扫 `links[].label` 是否含"回到/返回/回去/回来/return to/go back";若含且该 link 无 `requires` 保护 → 记为 `[可疑][P2] 路径前提`

### 需人工/reviewer 检查的部分(语义,工具查不到)
链接文字只是冰山一角。更隐蔽的路径前提是**节点正文(`look`)里的暗含引用**:
- "你还记得赛琳在营地说的那句话"——但有的路径没去过营地 / 没见过赛琳
- "那道熟悉的海湾曲线映入眼帘"——但从另一条路来的玩家从没见过这个海湾

**reviewer 循环时对每个 in-degree ≥ 2 的节点问三个问题:**
1. 节点正文(`look`)是否引用了"只有某条来路才经历过"的事件/NPC/地点?
2. 链接文字(`label`)是否说"回到/返回"一个并非所有来路都经过的地方?
3. 如果答案是"是":在 `look` 函数里按 flag 分流这段内容,或给链接加 `requires`,或把这段内容挪进只有正确来路才能到的分支节点。

### 修法(三种,按情况选)
1. **加 `requires`**:链接加 `requires:(S)=>S.flags.visited_village`,没去过村庄的玩家不显示"回到村庄"(配 `showWhenLocked:true`+`lockHint` 可灰显)。
2. **`look` 函数分流**:正文写成 `look:(S)=> '…' + (S.flags.visited_village ? '\n你想起了村庄里的场景。' : '')`,没去过的玩家不看到这段。(模块化呈现器**不解析** `{{}}` 标记,条件叙述一律靠 `look` 函数。)
3. **改链接文字**:把"回到村庄"改成"前往村庄"(去掉"回"字的前提假设),然后在 village_entrance 用 `look:{first:'首次版', return:'重访版'}`(引擎按访问计数自动判定首次/重访)。

### 真实案例(《制图师的挽歌》,旧版审计实测发现的模式)
- `map_fragment`(5 入边):"回去看看村庄的情况"→ `village_entrance`(无 requires)。非主线玩家从未去过村庄。
- `first_shift`(9 入边!):"先回到北部海滩"→ `act1_end`(无 requires)。绕路来的玩家没去过海滩。

## 九、人物突变/消失/改名(Character Continuity Failures)

### 表现
- **突然出现**:一个从未被介绍的 NPC 突然出现在对话中("赛琳的弟弟卡明插了一句话"——卡明是谁?从没提过)。
- **突然消失**:一个活跃角色在某条路径后完全不再出现,没有任何解释(不是死了,是作者忘了)。
- **改名**:同一角色在不同场景用了不同名字("赛琳"/"塞琳"/"赛林"),或同一地点用了不同名("古庙"/"神殿"/"遗迹"指的是同一个地方)。
- **状态不一致**:角色在路径 A 受了伤,但在路径 B 汇流后完好无损;角色在某场景承诺了一件事,后面完全不提。

### 防护(三层)
1. **canon.md 角色表**(写作时):每个重要角色一行,包括名字(唯一拼写)、当前状态、声音速写。**每个 NPC 首次出场前必须在 canon 里有条目**——如果 canon 里没有,不要在正文里突然创造一个。
2. **reviewer 检查**(每 15/10 场景):
   - 新场景里出现的 NPC 名字,canon 里都有吗?(新角色必须先更新 canon)
   - 角色名拼写全场一致?(grep 变体)
   - 路径分叉后角色状态(活/死/伤/离开)是否在 `look` 函数里按 flag 分流?
3. **graph-audit**(审计时):暂时无法自动检测角色改名(这是语义问题);上面"路径前提"里的入度脚本能定位高入边节点,人工据此查"角色在某路径后的节点被无条件引用"的情况。

### 修法
- 改名:全文搜替,统一为 canon 里的唯一名字。
- 突然出现:前置节点加铺垫(或在 `look` 函数里用 `S.flags.met_X` 条件引用)。
- 突然消失:在后续节点加条件引用("赛琳不在这里"或"赛琳上次说的话你还记得")。
- 状态不一致:在汇流节点的 `look` 函数里按 `S.flags.X_状态` 分流描写。

唯一和本项目引擎绑定的是 `look` 函数(读 `S.flags`)的条件叙述写法和 `graph-audit.mjs` 的具体命令。如果用别的引擎(Twine/Ink),原理一样,具体语法换掉即可。
